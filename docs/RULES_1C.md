# Правила разработки 1С (запросы, отчеты, работа с платформой)

Этот документ собран по практикам из книг, лежащих в `docs/books/`, и дополняется стандартами разработки 1С (ИТС).

## 0) Быстрый «скелет» решения (как действовать в реальной задаче)

- **Если нужно показать данные пользователю** → сначала думайте про **отчет на СКД** (настройки/варианты/расшифровка), а не про «таблицу значений и ручную печать».
- **Если нужно подготовить выборку** → сначала пишите **запрос**, проверяйте объём/края/типы, потом подключайте в СКД как набор данных.
- **Если нужно ускорить** → отборы и сокращение выборки «как можно раньше», параметры вместо литералов, минимизация соединений, диагностика на боевом объеме.

## 1) Язык запросов 1С: базовые правила

Основа: книга **Е.Ю. Хрусталева — «Язык запросов “1С:Предприятия 8”»** (`Radchenko_2013_Basics_1C83.pdf`).

- **Начинайте с модели данных**: перед написанием запроса четко зафиксируйте, *какие* объекты/таблицы метаданных являются источниками (справочники, документы, регистры) и *какие* поля реально нужны.
- **Выбирайте только нужные поля**: не делайте «выбрать всё», особенно для отчетов/форм — это бьет по сети/памяти и ухудшает планы выполнения.
- **Режьте объем данных как можно раньше**:
  - отборы (ГДЕ) по индексируемым полям/измерениям — как можно ближе к источнику;
  - используйте параметры запроса вместо «склейки строк» запроса.
- **Стабильно работайте с типами**:
  - осознанно приводите типы в выражениях (когда есть риск сравнения разных типов/неявных преобразований);
  - внимательно к `NULL`/пустым ссылкам и результатам соединений.
- **Соединения делайте осмысленно**:
  - минимизируйте количество соединяемых таблиц;
  - LEFT JOIN используйте только если действительно нужно «сохранить» левую часть при отсутствии правой.
- **Проверяйте запрос на крайних случаях**: пустой набор, единичная запись, большие объемы, «дырявые» данные (Нет значения, пометки удаления, отсутствующие связи).

### 1.1) Шаблон оформления текста запроса (универсальный)

Опирайтесь на стандарт оформления текста запроса (ИТС):
- `https://its.1c.ru/db/v8std/content/437/hdoc`

Рекомендуемый шаблон (как базовый стиль в проекте):

```bsl
ТекстЗапроса =
"ВЫБРАТЬ
|   // измерения/ключи
|   Т.Ссылка                                  КАК Ссылка,
|   Т.Наименование                            КАК Наименование,
|
|   // показатели
|   СУММА(Т.СуммаДокумента)                   КАК СуммаДокумента
|
|ИЗ
|   Документ.РеализацияТоваровУслуг КАК Т
|
|ГДЕ
|   Т.Дата МЕЖДУ &ДатаНач И &ДатаКон
|
|СГРУППИРОВАТЬ ПО
|   Т.Ссылка,
|   Т.Наименование
|
|УПОРЯДОЧИТЬ ПО
|   Наименование";
```

Правила к шаблону:
- **одна секция — один блок строк** (между секциями пустая строка);
- **алиасы (КАК …) выравнивайте**, чтобы глазами «сканировать» поля;
- **все параметры** — только как `&ИмяПараметра`, значения задаются через `Запрос.УстановитьПараметр()`.

### 1.2) Параметры запроса: как делать правильно

- **Не конкатенируйте значения в текст запроса** (кроме управляемых условных вставок по стандарту, если без этого никак).
- **Всегда используйте параметры** (`&...`) для дат, ссылок, списков значений и т. п.
- **Проверяйте пустой результат** стандартным способом (ИТС):
  - `https://its.1c.ru/db/v8std/content/438/hdoc`

Практический паттерн:

```bsl
Запрос = Новый Запрос(ТекстЗапроса);
Запрос.УстановитьПараметр("ДатаНач", ДатаНач);
Запрос.УстановитьПараметр("ДатаКон", ДатаКон);

Выборка = Запрос.Выполнить().Выбрать();
Если Выборка.Пустой() Тогда
    // корректный UX: показать "нет данных" и выйти
    Возврат;
КонецЕсли;
```

### 1.3) Частые антипаттерны в запросах (и чем заменить)

- **Антипаттерн: “ВЫБРАТЬ *”**
  - **Вместо этого**: перечисляйте только реально используемые поля.
- **Антипаттерн: строковые литералы, которые увидит пользователь**
  - **Вместо этого**: параметр `&Текст...` + локализуемое значение `НСтр(...)` (см. ИТС по локализации).
- **Антипаттерн: тяжёлые выражения в условиях**
  - **Вместо этого**: делайте условия «эффективными» (ИТС, пример точки входа):
    - `https://its.1c.ru/db/content/v8std/src/300/200/i8100658.htm`

## 3) Отчеты и СКД (Система компоновки данных)

Основа: книга **Е.Ю. Хрусталева — «Разработка сложных отчетов… СКД»** (`Khrustaleva_E_Yu_-_Razrabotka_slozhnykh_otchetov_v_1S_Predpriatii_8__-_2016.pdf`).

- **Стройте отчет от потребностей пользователя**:
  - какие разрезы (группировки) нужны;
  - какие показатели (ресурсы) нужны;
  - какие параметры пользователь должен менять сам, а какие фиксированы логикой.
- **Наборы данных**:
  - начинайте с одного набора (обычно «Запрос»), усложняйте только при необходимости;
  - при нескольких наборах данных явно продумывайте **связи наборов**.
- **Параметры и отборы**:
  - параметры периода и ключевые отборы должны быть явными и контролируемыми;
  - если значение параметра определяется прикладной логикой, задавайте его в модуле отчета в событиях компоновки (а не «прячьте» в выражениях).
- **Варианты отчета и пользовательские настройки**:
  - отделяйте «варианты поставки» от пользовательских вариантов;
  - используйте хранилища вариантов/настроек там, где это оправдано.
- **Вычисляемые поля и выражения**:
  - используйте вычисления там, где это реально логика представления/агрегации;
  - если вычисление сложно тестировать/поддерживать — лучше вынести подготовку данных в запрос/набор данных или в программную компоновку.
- **Расшифровка**:
  - проектируйте расшифровку как часть UX отчета: что именно пользователь должен «проваливать», какие поля являются ключами.

### 3.1) Мини‑гайд по СКД: как «собрать» отчет, который не стыдно поддерживать

- **Данные**
  - 1 набор данных‑запрос → пока не доказана необходимость нескольких.
  - поля набора данных называйте так, чтобы потом не «расшифровывать» их смысл в выражениях.
- **Параметры**
  - параметры периода/организации/отборов выводите как **быстрые пользовательские настройки**, если пользователю это нужно менять.
  - «технические» параметры (например, режим/флаги) держите скрытыми и задавайте программно.
- **Выражения**
  - если выражение влияет на интерфейс (заголовки, представления) — держите его простым и локализуемым;
  - сложную логику лучше вынести в подготовку данных.
- **Варианты**
  - минимум 1 «основной» вариант (поставляемый);
  - отдельные варианты под роли пользователей (бухгалтер/менеджер/руководитель), чтобы не перегружать настройками один вариант.
- **Расшифровка**
  - для каждого ключевого ресурса/показателя заранее определите: *куда* проваливаемся (форма списка? форма объекта? другой отчет?).

### 3.2) Когда нужно программно вмешиваться в СКД (а не только настройками)

Типовые случаи:
- значения параметров зависят от прикладной логики/прав/контекста;
- нужно подменить набор данных (например, разные источники по режиму);
- нужно динамически менять заголовки/оформление по параметрам.

Подход: вмешательство делайте **централизованно** (в модуле отчета, в «точках» компоновки), а не разносите по формам/командам.

См. стандарт ИТС:  
`https://its.1c.ru/db/v8std/content/762/hdoc`

## 5) Практика работы с платформой и разработкой конфигурации

Основа: книга **М.Г. Радченко, Е.Ю. Хрусталева — «1С:Предприятие 8.3. Практическое пособие разработчика…»** (`M_G_Radchenko_1SPredpriyatie_83_Prakticheskoe_posobie_razrabotchika.pdf`).

- **Разрабатывайте «от метаданных к поведению»**:
  - сначала корректно спроектируйте объекты (справочники/документы/регистры);
  - затем формы/командный интерфейс;
  - затем логику проведения/заполнения/контролей.
- **Разделяйте клиент и сервер**:
  - вычисления/доступ к данным — по возможности на сервере;
  - на клиенте оставляйте UI-логику, реакции на события формы и «склейку» представлений.
- **Отладка и синтакс‑помощник — обязательный навык**:
  - при ошибках сначала воспроизводимость и минимальный сценарий;
  - затем точки останова/просмотр значений/трассировка;
  - затем уточнение в синтакс‑помощнике по объектам/методам/событиям, которые участвуют.

## 7) Мини-чеклист перед сдачей (запросы/отчеты)

- **Запрос**
  - отборы/параметры есть, строковая «склейка» минимальна;
  - выводятся только нужные поля;
  - проверены крайние случаи (пусто/много/битые связи);
  - планируется адекватное время выполнения на «боевом» объеме.
- **СКД**
  - понятные параметры и дефолты;
  - группировки/ресурсы соответствуют постановке;
  - расшифровка работает и ведет к нужным формам/данным;
  - интерфейсные тексты локализуемы (см. ИТС `#std762`).

## 9) Рецепты запросов (шаблоны под реальные задачи)

Ниже — заготовки, которые удобно копировать и адаптировать. Они не «универсальны на 100%», но покрывают 80% типовых задач отчетов и обработок.

### 9.1) Остатки на дату (регистр накопления)

```bsl
ТекстЗапроса =
"ВЫБРАТЬ
|   Остатки.Номенклатура                           КАК Номенклатура,
|   Остатки.Склад                                 КАК Склад,
|   Остатки.КоличествоОстаток                      КАК Количество,
|   Остатки.СтоимостьОстаток                       КАК Стоимость
|ИЗ
|   РегистрНакопления.ТоварыНаСкладах.Остатки(&Дата, ) КАК Остатки
|ГДЕ
|   Остатки.Номенклатура В (&НоменклатураСписок)";

Запрос = Новый Запрос(ТекстЗапроса);
Запрос.УстановитьПараметр("Дата", ДатаНа);
Запрос.УстановитьПараметр("НоменклатураСписок", НоменклатураСписок); // Массив/СписокЗначений/ТЗ-колонка
```

Подсказки:
- если список может быть пустым — заранее решайте UX (показывать всё / не показывать ничего / требовать заполнения).
- для больших списков подумайте о временной таблице/таблице значений и соединении, чтобы не «раздувать» IN.

### 9.2) Обороты за период (регистр накопления)

```bsl
ТекстЗапроса =
```

### 9.4) Пакет запросов (когда нужно несколько наборов разом)

Идея: 1 текст, несколько `;`‑разделённых запросов, 1 выполнение → массив результатов.

```bsl
ТекстПакета =
"ВЫБРАТЬ
|   ...
|;
|
|ВЫБРАТЬ
|   ...";

Запрос = Новый Запрос(ТекстПакета);
// Запрос.УстановитьПараметр(...)

Результаты = Запрос.ВыполнитьПакет();
Выборка1 = Результаты[0].Выбрать();
Выборка2 = Результаты[1].Выбрать();
```

### 9.5) Временная таблица (чтобы “присоединить список” и не раздувать `В (...)`)

Используйте, когда:
- список значений большой (сотни/тысячи);
- нужно присоединять к нему дополнительные колонки (например, “минимальная цена”, “приоритет”);
- вы хотите контролировать план запроса и избегать огромного `IN`.

```bsl
ТекстЗапроса =
"ВЫБРАТЬ
|   Список.Номенклатура КАК Номенклатура
|ПОМЕСТИТЬ ВТНоменклатура
|ИЗ
|   &ТЗНоменклатура КАК Список
|;
|
|ВЫБРАТЬ
|   Остатки.Номенклатура                 КАК Номенклатура,
|   Остатки.КоличествоОстаток            КАК Количество
|ИЗ
|   РегистрНакопления.ТоварыНаСкладах.Остатки(&Дата, ) КАК Остатки
|       ВНУТРЕННЕЕ СОЕДИНЕНИЕ ВТНоменклатура КАК ВТ
|       ПО Остатки.Номенклатура = ВТ.Номенклатура";

Запрос = Новый Запрос(ТекстЗапроса);
Запрос.УстановитьПараметр("Дата", ДатаНа);
Запрос.УстановитьПараметр("ТЗНоменклатура", ТЗНоменклатура); // ТаблицаЗначений с колонкой Номенклатура
```

### 9.6) “Разворот” (pivot) по типу/статусу через `ВЫБОР ... СУММА(...)`

Приём: превращаем строки разных типов в отдельные колонки “Приход/Расход/Возврат/…” без постобработки.

```bsl
ТекстЗапроса =
"ВЫБРАТЬ
|   Движения.Номенклатура                                        КАК Номенклатура,
|   СУММА(ВЫБОР КОГДА Движения.ВидДвижения = &Приход  ТОГДА Движения.Количество ИНАЧЕ 0 КОНЕЦ) КАК КолПриход,
|   СУММА(ВЫБОР КОГДА Движения.ВидДвижения = &Расход  ТОГДА Движения.Количество ИНАЧЕ 0 КОНЕЦ) КАК КолРасход
|ИЗ
|   РегистрНакопления.ТоварыНаСкладах.Обороты(&ДатаНач, &ДатаКон, , ) КАК Движения
|СГРУППИРОВАТЬ ПО
|   Движения.Номенклатура";

Запрос = Новый Запрос(ТекстЗапроса);
Запрос.УстановитьПараметр("ДатаНач", ДатаНач);
Запрос.УстановитьПараметр("ДатаКон", ДатаКон);
Запрос.УстановитьПараметр("Приход", ВидДвиженияНакопления.Приход);
Запрос.УстановитьПараметр("Расход", ВидДвиженияНакопления.Расход);
```

## 10) Рецепты отчетов на СКД (практичные схемы)

### 10.1) «Таблица + Диаграмма» в одном отчете

Подход:
- один набор данных;
- два вывода/две области (табличный документ + диаграмма) или два варианта отчета;
- диаграмму строить по тем же ресурсам/группировкам, что и таблицу, чтобы не было «двух разных отчетов».

Чек:
- ресурсы диаграммы = ресурсы таблицы (или явно объяснено, почему отличаются);
- параметры (период/отбор) общие;
- в заголовках/подписях никаких «зашитых» строк, если они выходят пользователю (см. #std762).

### 10.2) Детализация и «проваливание»

Рекомендуемый путь:
- в основной таблице выводите агрегаты и «ключи» расшифровки;
- расшифровка по клику ведёт:
  - либо на форму списка с отбором,
  - либо на подчинённый отчет‑детализацию,
  - либо на форму объекта (если смысл именно в карточке).

Технический принцип:
- поле, по которому проваливаемся, должно быть **ссылкой/ключом**, а не строковым представлением.

### 10.3) Типовой отчет СКД «Остатки + Обороты + Конечный остаток»

Задача: дать пользователю картину по ресурсу (количество/сумма) за период, с начальным остатком, оборотами и конечным остатком.

Рекомендуемая схема:
- **Параметры**:
  - `ДатаНач`, `ДатаКон` (период),
  - (опционально) `Склад`, `Организация`, `Номенклатура` (отборы).
- **Набор данных**: один запрос на базе виртуальной таблицы **Обороты** регистра, который возвращает:
  - `ОстатокНачальный`,
  - `Приход`,
  - `Расход`,
  - `ОстатокКонечный`.

Ключевая идея: не считать конечный остаток “вручную” на клиенте, а получать его из механизма регистра/виртуальных таблиц.

Скелет запроса (примерно):

```bsl
ТекстЗапроса =
"ВЫБРАТЬ
|   Данные.Номенклатура                          КАК Номенклатура,
|   Данные.Склад                                КАК Склад,
|
|   Данные.КоличествоНачальныйОстаток            КАК НачОстаток,
|   Данные.КоличествоПриход                      КАК Приход,
|   Данные.КоличествоРасход                      КАК Расход,
|   Данные.КоличествоКонечныйОстаток             КАК КонОстаток
|ИЗ
|   РегистрНакопления.ТоварыНаСкладах.Обороты(&ДатаНач, &ДатаКон, , ) КАК Данные
|ГДЕ
|   (&Склад ЕСТЬ NULL ИЛИ Данные.Склад = &Склад)";
```

Настройка СКД:
- **Группировки**: Номенклатура → (опц.) Склад.
- **Ресурсы**: НачОстаток, Приход, Расход, КонОстаток.
- **Расшифровка**:
  - по Номенклатуре/Складу — проваливание в список движений/документов (отчет‑детализация или форма списка документов с отбором по периоду и аналитикам).

## 11) Как работать с синтакс‑помощником (памятка)

Цель — быстро находить **правильные типы/методы/события** и не гадать по памяти.

- **Ищите по “классу” объекта, а не по “как я его назвал в переменной”**:
  - пример: переменная `Выборка` → ищите “Выборка из результата запроса” / “Результат запроса”.
- **Если не знаете, какой это тип**:
  - временно выводите тип в отладке/через `ТипЗнч()` (на время диагностики),
  - затем ищите по названию типа в синтакс‑помощнике.
- **Для UI‑событий ищите по объекту формы/элемента**:
  - “УправляемаяФорма.ПриСозданииНаСервере”,
  - “Расширение поля формы …” (как вы уже видите в комментариях проекта).
- **Для запросов ключевые сущности**:
  - `Запрос` (установка параметров, выполнение),
  - результат выполнения (`Выполнить`, `ВыполнитьПакет`),
  - выборка (`Выбрать`, `Следующий`, `Пустой`).

## 12) Лучшие проверенные практики (ИТС v8std) — «что делать всегда»

Ниже — то, что реально помогает в больших конфигурациях: читабельность, предсказуемость поведения и производительность.

### 12.1) Запросы: базовый минимум качества

- **Оформляйте текст запроса по стандарту** (#std437): `https://its.1c.ru/db/v8std/content/437/hdoc`
- **Проверяйте пустой результат правильно** (#std438): `https://its.1c.ru/db/v8std/content/438/hdoc`
- **Делайте условия эффективными** (точка входа по условиям): `https://its.1c.ru/db/content/v8std/src/300/200/i8100658.htm`
- **Не плодите повторные выполнения однотипных запросов** (#std436): `https://its.1c.ru/db/v8std/content/436/hdoc`

### 12.2) Запросы: «опасные зоны» (то, что нужно избегать без крайней необходимости)

Список ниже — типовые причины тормозов и неожиданных результатов:

- **ПОЛНОЕ ВНЕШНЕЕ СОЕДИНЕНИЕ** — используйте только когда иначе нельзя (#std435): `https://its.1c.ru/db/v8std/content/435/hdoc`
- **ОБЪЕДИНИТЬ / ОБЪЕДИНИТЬ ВСЕ** — осознанно, с пониманием последствий (#std434): `https://its.1c.ru/db/v8std/content/434/hdoc`
- **УПОРЯДОЧИТЬ ПО** — не сортируйте «на всякий случай» (#std412): `https://its.1c.ru/db/v8std/content/412/hdoc`
- **ПОДОБНО (LIKE)** — учитывайте особенности и влияние на производительность (#std726): `https://its.1c.ru/db/v8std/content/726/hdoc`
- **Псевдонимы источников** — делайте псевдонимы осмысленными и единообразными (#std758): `https://its.1c.ru/db/v8std/content/758/hdoc`
- **COUNT и “сколько строк”** — делайте корректно, без лишней нагрузки (#std787): `https://its.1c.ru/db/v8std/content/787/hdoc`

### 12.3) Локализация: запросы/динамические списки/СКД

- **Никаких “зашитых” строк**, которые видит пользователь: выносите в параметры и задавайте локализуемо (#std762): `https://its.1c.ru/db/v8std/content/762/hdoc`

### 12.4) Встроенный язык: базовая дисциплина кода

- Точка входа по общим требованиям к конструкциям встроенного языка: `https://its.1c.ru/db/v8std/content/441/hdoc`
- **Всегда разделяйте ответственность**: данные/бизнес‑логика/представление (клиент) — не смешивайте в одном обработчике «всё сразу».
- **Пишите код так, чтобы его можно было отлаживать**:
  - минимум скрытой динамики (склейки строк, “магии” с неявными типами);
  - предсказуемые точки принятия решений (ранние `Возврат`, явные проверки).

### 12.5) СКД: практики поддерживаемых отчетов

- **Один набор данных** до тех пор, пока не доказана необходимость нескольких.
- **Варианты отчета**:
  - поставляемые варианты — минимальный набор «по ролям»;
  - пользовательские варианты — отдельно, без “ломания” основного.
- **Расшифровка = часть UX**:
  - проваливание должно идти по ссылкам/ключам, а не по представлениям;
  - сценарий «сверху вниз» продуман (от агрегата → к документу/записи).

## 13) Контрольный список «перед тем как сказать "готово"»

- **Запрос**

  - расшифровка ведёт к ожидаемым данным;

## 14) Транзакции и блокировки (ИТС)

### 14.1) Основные правила работы с транзакциями

См. ИТС:
- Управление транзакциями: `https://its.1c.ru/db/v8std/content/452/hdoc`
- Блокировка данных: `https://its.1c.ru/db/v8std/content/453/hdoc`

**Ключевые принципы:**
- **Минимизируйте время транзакции** — открывайте как можно позже, закрывайте как можно раньше;
- **Не вызывайте интерактивные методы внутри транзакции** (`Предупреждение`, `Вопрос`, `Сообщить`);
- **Не выполняйте длительные вычисления внутри транзакции**;
- **Используйте управляемые блокировки** для оптимистичного контроля:

```bsl
Блокировка = Новый БлокировкаДанных;
Элемент = Блокировка.Добавить("РегистрНакопления.ТоварыНаСкладах");
Элемент.Режим = РежимБлокировкиДанных.Исключительный;
Элемент.УстановитьЗначение("Склад", Склад);
Блокировка.Заблокировать();

НачатьТранзакцию();
Попытка
    // запись данных
    ЗафиксироватьТранзакцию();
Исключение
    ОтменитьТранзакцию();
    ВызватьИсключение;
КонецПопытки;
```

### 14.2) Обработка ошибок и транзакции

**Правило:** всегда используйте конструкцию `Попытка...Исключение` внутри транзакций:

```bsl
НачатьТранзакцию();
Попытка
    // операции с данными
    Записать();
    ЗафиксироватьТранзакцию();
Исключение
    ОтменитьТранзакцию();
    ЗаписьЖурналаРегистрации(
        "ЗаписьДокумента", 
        УровеньЖурналаРегистрации.Ошибка,
        ,
        ,
        ОписаниеОшибки()
    );
    ВызватьИсключение;
КонецПопытки;
```

## 15) Управляемые формы: стандарты и практики (ИТС)

### 15.1) Основные принципы

См. ИТС:
- Построение форм: `https://its.1c.ru/db/v8std/content/573/hdoc`
- Клиент-серверное взаимодействие: `https://its.1c.ru/db/v8std/content/574/hdoc`

**Ключевые правила:**
- **Разделяйте клиентский и серверный код**:
  - `&НаКлиенте` — только UI-логика, реакции на изменения, вызов сервера;
  - `&НаСервере` / `&НаСервереБезКонтекста` — работа с данными, вычисления;
- **Минимизируйте передачу данных между клиентом и сервером**;
- **Используйте `&НаСервереБезКонтекста`** где возможно (быстрее, не требует передачи контекста формы);

```bsl
&НаКлиенте
Процедура КомандаВыполнить(Команда)
    // Только вызов серверного метода и обработка результата
    Результат = СервернаяФункция(Параметр);
    Если Результат Тогда
        Оповестить("ДанныеОбновлены");
    КонецЕсли;
КонецПроцедуры

&НаСервереБезКонтекста
Функция СервернаяФункция(Параметр)
    // Работа с данными без контекста формы
    Запрос = Новый Запрос(ТекстЗапроса);
    Запрос.УстановитьПараметр("Параметр", Параметр);
    Результат = Запрос.Выполнить();
    Возврат Не Результат.Пустой();
КонецФункции
```

### 15.2) Динамические списки

См. ИТС: `https://its.1c.ru/db/v8std/content/580/hdoc`

**Рекомендации:**
- Используйте динамические списки для больших объемов данных (автоматическая подкачка);
- Настраивайте отборы на сервере через `ОбработкаПолученияДанных`:

```bsl
&НаСервере
Процедура СписокПриПолученииДанных(Элемент, Отказ, СтандартнаяОбработка)
    Если СтандартнаяОбработка Тогда
        // Дополнительная обработка запроса
        Запрос = Элемент.ПолучитьЗапрос();
        ТекстЗапроса = Запрос.Текст;
        // Модификация запроса
        Запрос.Текст = ТекстЗапроса;
        Элемент.УстановитьЗапрос(Запрос);
    КонецЕсли;
КонецПроцедуры
```

### 15.3) Команды и действия форм

- **Команды формы** — группируйте логически (основные, дополнительные);
- **Глобальные команды** — только действительно глобальные действия;
- **Контекстные команды** — привязывайте к конкретным элементам;
- **Используйте `ПараметрыКоманды`** для передачи данных в обработчик.

## 16) Регистры: проектирование и использование

### 16.1) Регистры накопления

См. ИТС: `https://its.1c.ru/db/v8std/content/239/hdoc`

**Ведущие измерения:**
- Включайте для автоматического контроля актуальности;

## 17) Документы: проведение и движения

### 17.1) Модели проведения

См. ИТС: `https://its.1c.ru/db/v8std/content/252/hdoc`

**Три модели:**
1. **Оперативная** (по умолчанию) — движения в момент записи:
   ```bsl
   Процедура ОбработкаПроведения(Отказ, Режим)
       // Формирование движений
   КонецПроцедуры
   ```

2. **Оперативная с проведением** — разделение записи и проведения:
   ```bsl
   Процедура ОбработкаПроведения(Отказ, РежимПроведения)
       Если РежимПроведения.РежимПроведения = РежимПроведенияДокумента.Оперативный Тогда
           // Оперативное проведение
       КонецЕсли;
   КонецПроцедуры
   ```

3. **Перепроведение** — для пакетных операций:
   ```bsl
   Процедура ОбработкаПроведения(Отказ, РежимПроведения)
       Если РежимПроведения.РежимПроведения = РежимПроведенияДокумента.Перепроведение Тогда
           // Удаление старых движений
           Движения.Очистить();
       КонецЕсли;
       // Формирование новых движений
   КонецПроцедуры
   ```

### 17.2) Движения документов

**Правила:**
- **Заполняйте регистратор** — ссылка на документ автоматически;
- **Контролируйте отрицательные остатки**:
  ```bsl
  Процедура ОбработкаПроведения(Отказ, Режим)
      Режим.ДопускатьОтрицательныеОстатки = Ложь;
      // Движения
  КонецПроцедуры
  ```
- **Используйте `Движения.Очистить()`** при перепроведении.

1. **Отборы на индексных полях** — всегда в первую очередь;
2. **Временные таблицы для больших списков** — вместо `В (...)`;
3. **Не используйте `ЕСТЬ NULL` без необходимости** — лучше `&Параметр ЕСТЬ NULL ИЛИ Поле = &Параметр`;
4. **Избегайте функций в условиях WHERE** — `ГДЕ Год(Дата) = 2024` вместо `ГДЕ Дата МЕЖДУ ...`;
5. **Используйте `ТОЛЬКО ОБЪЕКТЫ`** для ограничений прав:
   ```bsl
   "ВЫБРАТЬ
   |   Поле1, Поле2
   |ИЗ
   |   Документ.Реализация КАК Т
   |ГДЕ
   |   Т.Ссылка В (ВЫБРАТЬ Ссылка ИЗ Документ.Реализация ТОЛЬКО ОБЪЕКТЫ)";
   ```

### 20.2) Работа с большими объемами данных

**Пакетная обработка:**
```bsl
&НаСервере
Процедура ПакетнаяОбработка()
    Запрос = Новый Запрос(ТекстЗапроса);
    Выборка = Запрос.Выполнить().Выбрать();
    
    Пакет = 100;
    Счетчик = 0;
    НачатьТранзакцию();
    
    Пока Выборка.Следующий() Цикл
        Счетчик = Счетчик + 1;

        Если Счетчик % Пакет = 0 Тогда
            ЗафиксироватьТранзакцию();
            НачатьТранзакцию();
        КонецЕсли;
    КонецЦикла;
    
    Если Не ТранзакцияАктивна() Тогда
        НачатьТранзакцию();
    КонецЕсли;
```

# Стандарты разработки на основе БСП (Библиотека Стандартных Подсистем)

> Раздел основан на анализе БСП редакции 3.1 (версия 3.1.10.441)

## 25) Модульная архитектура: система суффиксов

БСП использует строгую систему именования общих модулей по суффиксам. Это обеспечивает:
- Понятное назначение каждого модуля
- Корректное разделение клиент/сервер
- Возможность расширения без модификации ядра

### 25.1) Суффиксы модулей

| Суффикс | Назначение | Флаги | Пример |
|---------|-----------|-------|--------|
| *(без суффикса)* | Основной серверный API подсистемы | Сервер, ВнешнееСоединение, ВызовСервера | `Пользователи`, `УправлениеДоступом`, `РаботаСФайлами` |
| `Служебный` | Внутренняя реализация, не для внешних вызовов | Сервер | `ВариантыОтчетовСлужебный`, `АдресныйКлассификаторСлужебный` |
| `СлужебныйКлиент` | Служебный код на клиенте | Клиент | `ВариантыОтчетовСлужебныйКлиент` |
| `СлужебныйКлиентСервер` | Служебный код общего назначения | Клиент, Сервер | `ВариантыОтчетовСлужебныйКлиентСервер` |
| `СлужебныйВызовСервера` | Обёртки для серверных вызовов из служебного кода | Сервер, ВызовСервера | `ВариантыОтчетовСлужебныйВызовСервера` |
| `СлужебныйПовтИсп` | Служебный модуль повторного использования | Сервер | `АдресныйКлассификаторПовтИсп` |
| `Клиент` | Клиентская часть подсистемы | Клиент | `ВариантыОтчетовКлиент`, `УправлениеПечатьюКлиент` |
| `КлиентСервер` | Модуль общего назначения (клиент + сервер) | Клиент, Сервер | `СтроковыеФункцииКлиентСервер`, `ДлительныеОперацииКлиентСервер` |
| `ВызовСервера` | Обёртки серверных вызовов для клиентского кода | Сервер, ВызовСервера | `БизнесПроцессыИЗадачиВызовСервера` |
| `Переопределяемый` | Модуль с хуками для расширения другими разработчиками | Сервер, ВнешнееСоединение | `УправлениеДоступомПереопределяемый` |
| `КлиентПереопределяемый` | Клиентские переопределяемые процедуры | Клиент | `БизнесПроцессыИЗадачиКлиентПереопределяемый` |
| `Глобальный` | Глобальный модуль (доступен без префикса) | Глобальный | `ЦентрМониторингаГлобальный` |
| `Локализация` | Региональная локализация (РФ) | Клиент, Сервер | `СтроковыеФункцииКлиентСерверЛокализация` |

### 25.2) Типовой набор модулей подсистемы

Для каждой подсистемы БСП создаёт 5–12 модулей:

```
<Подсистема>                     — серверный API (экспортные процедуры/функции)
<Подсистема>Клиент               — клиентская логика (команды формы, реакции на события)
<Подсистема>КлиентСервер         — общие функции, доступные и клиенту, и серверу
<Подсистема>ВызовСервера         — обёртки серверных вызовов для клиента
<Подсистема>Служебный            — внутренняя реализация (не экспортный)
<Подсистема>СлужебныйКлиент      — клиентские служебные процедуры
<Подсистема>СлужебныйКлиентСервер — общие служебные функции
<Подсистема>СлужебныйВызовСервера — служебные серверные вызовы
<Подсистема>СлужебныйПовтИсп     — код для переиспользования другими подсистемами
<Подсистема>Переопределяемый     — хуки: пустые процедуры для переопределения
<Подсистема>КлиентПереопределяемый — клиентские хуки
```

**Правило:** не создавайте «god-модули» — логику разделяйте по назначению и по клиент/сервер.

### 25.3) Переопределяемые модули (механизм хуков)

БСП позволяет расширять поведение без модификации ядра через переопределяемые модули:

```bsl

Процедура ПослеПроверкиВключенияПользователяВГруппу(Пользователь, Группа) Экспорт
    // Пустая реализация — переопределяется в прикладном решении
КонецПроцедуры

// Переопределяемая функция: позволяет добавить дополнительные ограничения
Функция ПроверитьДополнительныеОграничения(Пользователь, Объект) Экспорт
    Возврат Истина; // По умолчанию — без ограничений
КонецФункции
```

**Правило:** если ваша конфигурация расширяет БСП, создавайте свои переопределяемые модули по тому же паттерну.

# 39) Антипаттерны и известные ошибки генерации XML метаданных

> Раздел создан на основе реальных ошибок, допущенных при генерации конфигурации для платформы 1С:Предприятие 8.3.27.
> **Всегда проверять эти пункты перед записью XML-файлов.**

## 39.1) Формы (Form)

**Ошибка:** Добавление свойств `ExtendedPresentation` и `Explanation` в объект `Form`.
**Правило:** Эти свойства **не входят** в состав метаданных формы. Удалять их при генерации XML форм.

```xml
<!-- НЕПРАВИЛЬНО -->
<Form ...>
    <Properties>
        ...
        <ExtendedPresentation/>
        <Explanation/>
    </Properties>
</Form>

<!-- ПРАВИЛЬНО -->
<Form ...>
    <Properties>
        <Name>...</Name>
        <Synonym>...</Synonym>
        <FormType>Managed</FormType>
        <!-- ExtendedPresentation и Explanation ОТСУТСТВУЮТ -->
    </Properties>
</Form>
```

## 39.2) Табличные части справочников (Catalog.TabularSection)

**Ошибка:** Отсутствие узла `InternalInfo` для табличной части в справочнике.
**Правило:** Платформа 8.3.27 требует наличия `InternalInfo` с типами `CatalogTabularSection` и `CatalogTabularSectionRow` для каждой табличной части справочника.

```xml
<!-- ПРАВИЛЬНО -->
<TabularSection uuid="...">
    <InternalInfo>
        <xr:GeneratedType name="CatalogTabularSection.Справочник.ИмяТЧ" category="TabularSection">
            <xr:TypeId>...</xr:TypeId>
            <xr:ValueId>...</xr:ValueId>
        </xr:GeneratedType>
        <xr:GeneratedType name="CatalogTabularSectionRow.Справочник.ИмяТЧ" category="TabularSectionRow">
            <xr:TypeId>...</xr:TypeId>
            <xr:ValueId>...</xr:ValueId>
        </xr:GeneratedType>
    </InternalInfo>
    <Properties>...</Properties>
</TabularSection>
```

## 39.3) Атрибуты табличных частей (TabularSection.Attribute)

**Ошибка:** Добавление свойств `FillValue`, `FillFromFillingValue`, `ChoiceFoldersAndItems`, `Indexing` в атрибуты табличных частей.
**Правило:** Атрибуты ТЧ поддерживают только `Name`, `Synonym`, `Comment`, `Type`. Остальные свойства (заполнение, выбор, индексация) **недопустимы**.

```xml
<!-- НЕПРАВИЛЬНО -->
<Attribute uuid="...">
    <Properties>
        <Name>Поле</Name>
        <Type>...</Type>
        <FillValue xsi:type="xs:string"/> <!-- ОШИБКА -->
        <Indexing>Index</Indexing>        <!-- ОШИБКА -->
    </Properties>
</Attribute>

<!-- ПРАВИЛЬНО -->
<Attribute uuid="...">
    <Properties>
        <Name>Поле</Name>
        <Type>...</Type>
    </Properties>
</Attribute>
```

## 39.4) Справочники (Catalog)

**Ошибка:** Использование `xr:ObjectId` вместо `xr:ValueId` в `InternalInfo`.
**Правило:** Для версии платформы 8.3.27 использовать `xr:ValueId` для идентификаторов типов (Object, Ref, Selection, List, Manager).

## 39.5) План видов характеристик (ChartOfCharacteristicTypes)

**Ошибка:** Хранение свойств объектов в табличных частях справочников строковыми значениями (`ТипСвойства` как `xs:string`).
**Правило:** Для типизированных свойств использовать механизм `ChartOfCharacteristicTypes` (План видов характеристик) БСП:

```xml
<!-- ChartOfCharacteristicTypes.ВидыХарактеристик -->
<ChartOfCharacteristicTypes uuid="...">
    <Properties>
        <Name>ВидыХарактеристик</Name>
        <CharacteristicType>CatalogRef.Характеристики</CharacteristicType>
        <!-- Определяет, какой справочник хранит значения характеристик -->
    </Properties>
</ChartOfCharacteristicTypes>

<!-- Справочник Характеристики: вместо строки — ссылка на план видов -->
<Attribute uuid="...">
    <Properties>
        <Name>ВидХарактеристики</Name>
        <Type>
            <v8:Type>cfg:ChartOfCharacteristicTypesRef.ВидыХарактеристик</v8:Type>
        </Type>
    </Properties>
</Attribute>
```

**Архитектура свойств по БСП:**
| Объект | Назначение |
|---|---|
| `ChartOfCharacteristicTypes.ВидыХарактеристик` | Типы свойств: "Цвет" (строка), "Размер" (строка), "Вес" (число) |
| `Catalog.ЗначенияСвойствОбъектов` | Конкретные значения: "Красный", "XL", "5.2 кг" |
| `InformationRegister.ДополнительныеСведения` | Связь: Объект + Свойство → Значение |

## 39.6) Общие правила валидации

1. **Сверять структуру с примерами из папки `Examp`** перед генерацией нового объекта.
2. **Не копировать слепо свойства** из одного типа объекта в другой (например, свойства реквизита справочника не подходят для атрибута табличной части).
3. **Генерировать уникальные UUID** через PowerShell `[guid]::NewGuid()`, не использовать заглушки вроде `11111111-...` или `aaaaaaaa-...` для финальных файлов.
