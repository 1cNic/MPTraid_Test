# Правила разработки 1С (запросы, отчеты, работа с платформой)

Этот документ собран по практикам из книг, лежащих в `docs/books/`, и дополняется стандартами разработки 1С (ИТС).

## 0) Быстрый «скелет» решения (как действовать в реальной задаче)

- **Если нужно показать данные пользователю** → сначала думайте про **отчет на СКД** (настройки/варианты/расшифровка), а не про «таблицу значений и ручную печать».
- **Если нужно подготовить выборку** → сначала пишите **запрос**, проверяйте объём/края/типы, потом подключайте в СКД как набор данных.
- **Если нужно ускорить** → отборы и сокращение выборки «как можно раньше», параметры вместо литералов, минимизация соединений, диагностика на боевом объеме.

## 1) Язык запросов 1С: базовые правила

Основа: книга **Е.Ю. Хрусталева — «Язык запросов “1С:Предприятия 8”»** (`Radchenko_2013_Basics_1C83.pdf`).

- **Начинайте с модели данных**: перед написанием запроса четко зафиксируйте, *какие* объекты/таблицы метаданных являются источниками (справочники, документы, регистры) и *какие* поля реально нужны.
- **Выбирайте только нужные поля**: не делайте «выбрать всё», особенно для отчетов/форм — это бьет по сети/памяти и ухудшает планы выполнения.
- **Режьте объем данных как можно раньше**:
  - отборы (ГДЕ) по индексируемым полям/измерениям — как можно ближе к источнику;
  - используйте параметры запроса вместо «склейки строк» запроса.
- **Стабильно работайте с типами**:
  - осознанно приводите типы в выражениях (когда есть риск сравнения разных типов/неявных преобразований);
  - внимательно к `NULL`/пустым ссылкам и результатам соединений.
- **Соединения делайте осмысленно**:
  - минимизируйте количество соединяемых таблиц;
  - LEFT JOIN используйте только если действительно нужно «сохранить» левую часть при отсутствии правой.
- **Проверяйте запрос на крайних случаях**: пустой набор, единичная запись, большие объемы, «дырявые» данные (Нет значения, пометки удаления, отсутствующие связи).

### 1.1) Шаблон оформления текста запроса (универсальный)

Опирайтесь на стандарт оформления текста запроса (ИТС):
- `https://its.1c.ru/db/v8std/content/437/hdoc`

Рекомендуемый шаблон (как базовый стиль в проекте):

```bsl
ТекстЗапроса =
"ВЫБРАТЬ
|   // измерения/ключи
|   Т.Ссылка                                  КАК Ссылка,
|   Т.Наименование                            КАК Наименование,
|
|   // показатели
|   СУММА(Т.СуммаДокумента)                   КАК СуммаДокумента
|
|ИЗ
|   Документ.РеализацияТоваровУслуг КАК Т
|
|ГДЕ
|   Т.Дата МЕЖДУ &ДатаНач И &ДатаКон
|
|СГРУППИРОВАТЬ ПО
|   Т.Ссылка,
|   Т.Наименование
|
|УПОРЯДОЧИТЬ ПО
|   Наименование";
```

Правила к шаблону:
- **одна секция — один блок строк** (между секциями пустая строка);
- **алиасы (КАК …) выравнивайте**, чтобы глазами «сканировать» поля;
- **все параметры** — только как `&ИмяПараметра`, значения задаются через `Запрос.УстановитьПараметр()`.

### 1.2) Параметры запроса: как делать правильно

- **Не конкатенируйте значения в текст запроса** (кроме управляемых условных вставок по стандарту, если без этого никак).
- **Всегда используйте параметры** (`&...`) для дат, ссылок, списков значений и т. п.
- **Проверяйте пустой результат** стандартным способом (ИТС):
  - `https://its.1c.ru/db/v8std/content/438/hdoc`

Практический паттерн:

```bsl
Запрос = Новый Запрос(ТекстЗапроса);
Запрос.УстановитьПараметр("ДатаНач", ДатаНач);
Запрос.УстановитьПараметр("ДатаКон", ДатаКон);

Выборка = Запрос.Выполнить().Выбрать();
Если Выборка.Пустой() Тогда
    // корректный UX: показать "нет данных" и выйти
    Возврат;
КонецЕсли;
```

### 1.3) Частые антипаттерны в запросах (и чем заменить)

- **Антипаттерн: “ВЫБРАТЬ *”**
  - **Вместо этого**: перечисляйте только реально используемые поля.
- **Антипаттерн: строковые литералы, которые увидит пользователь**
  - **Вместо этого**: параметр `&Текст...` + локализуемое значение `НСтр(...)` (см. ИТС по локализации).
- **Антипаттерн: тяжёлые выражения в условиях**
  - **Вместо этого**: делайте условия «эффективными» (ИТС, пример точки входа):
    - `https://its.1c.ru/db/content/v8std/src/300/200/i8100658.htm`

## 3) Отчеты и СКД (Система компоновки данных)

Основа: книга **Е.Ю. Хрусталева — «Разработка сложных отчетов… СКД»** (`Khrustaleva_E_Yu_-_Razrabotka_slozhnykh_otchetov_v_1S_Predpriatii_8__-_2016.pdf`).

- **Стройте отчет от потребностей пользователя**:
  - какие разрезы (группировки) нужны;
  - какие показатели (ресурсы) нужны;
  - какие параметры пользователь должен менять сам, а какие фиксированы логикой.
- **Наборы данных**:
  - начинайте с одного набора (обычно «Запрос»), усложняйте только при необходимости;
  - при нескольких наборах данных явно продумывайте **связи наборов**.
- **Параметры и отборы**:
  - параметры периода и ключевые отборы должны быть явными и контролируемыми;
  - если значение параметра определяется прикладной логикой, задавайте его в модуле отчета в событиях компоновки (а не «прячьте» в выражениях).
- **Варианты отчета и пользовательские настройки**:
  - отделяйте «варианты поставки» от пользовательских вариантов;
  - используйте хранилища вариантов/настроек там, где это оправдано.
- **Вычисляемые поля и выражения**:
  - используйте вычисления там, где это реально логика представления/агрегации;
  - если вычисление сложно тестировать/поддерживать — лучше вынести подготовку данных в запрос/набор данных или в программную компоновку.
- **Расшифровка**:
  - проектируйте расшифровку как часть UX отчета: что именно пользователь должен «проваливать», какие поля являются ключами.

### 3.1) Мини‑гайд по СКД: как «собрать» отчет, который не стыдно поддерживать

- **Данные**
  - 1 набор данных‑запрос → пока не доказана необходимость нескольких.
  - поля набора данных называйте так, чтобы потом не «расшифровывать» их смысл в выражениях.
- **Параметры**
  - параметры периода/организации/отборов выводите как **быстрые пользовательские настройки**, если пользователю это нужно менять.
  - «технические» параметры (например, режим/флаги) держите скрытыми и задавайте программно.
- **Выражения**
  - если выражение влияет на интерфейс (заголовки, представления) — держите его простым и локализуемым;
  - сложную логику лучше вынести в подготовку данных.
- **Варианты**
  - минимум 1 «основной» вариант (поставляемый);
  - отдельные варианты под роли пользователей (бухгалтер/менеджер/руководитель), чтобы не перегружать настройками один вариант.
- **Расшифровка**
  - для каждого ключевого ресурса/показателя заранее определите: *куда* проваливаемся (форма списка? форма объекта? другой отчет?).

### 3.2) Когда нужно программно вмешиваться в СКД (а не только настройками)

Типовые случаи:
- значения параметров зависят от прикладной логики/прав/контекста;
- нужно подменить набор данных (например, разные источники по режиму);
- нужно динамически менять заголовки/оформление по параметрам.

Подход: вмешательство делайте **централизованно** (в модуле отчета, в «точках» компоновки), а не разносите по формам/командам.

См. стандарт ИТС:  
`https://its.1c.ru/db/v8std/content/762/hdoc`

## 5) Практика работы с платформой и разработкой конфигурации

Основа: книга **М.Г. Радченко, Е.Ю. Хрусталева — «1С:Предприятие 8.3. Практическое пособие разработчика…»** (`M_G_Radchenko_1SPredpriyatie_83_Prakticheskoe_posobie_razrabotchika.pdf`).

- **Разрабатывайте «от метаданных к поведению»**:
  - сначала корректно спроектируйте объекты (справочники/документы/регистры);
  - затем формы/командный интерфейс;
  - затем логику проведения/заполнения/контролей.
- **Разделяйте клиент и сервер**:
  - вычисления/доступ к данным — по возможности на сервере;
  - на клиенте оставляйте UI-логику, реакции на события формы и «склейку» представлений.
- **Отладка и синтакс‑помощник — обязательный навык**:
  - при ошибках сначала воспроизводимость и минимальный сценарий;
  - затем точки останова/просмотр значений/трассировка;
  - затем уточнение в синтакс‑помощнике по объектам/методам/событиям, которые участвуют.

## 5.1) Примитивные типы данных

Основа: книга **И.И. Низамутдинова — «Программирование в 1С за 11 шагов»**, глава 2.

Встроенный язык 1С поддерживает **динамическую типизацию** — тип определяется при присваивании. Переменная может менять тип в ходе выполнения.

### Основные примитивные типы

| Тип | Пример | Описание |
|-----|--------|----------|
| **Число** | `1500`, `3.14`, `-100` | Хранится с произвольной точностью. Квалификаторы: длина, точность |
| **Строка** | `"Товар"`, `'Кавычки внутри'` | Поддержка Unicode. Многострочные: `|"Строка1 \|` + `|Строка2"` |
| **Дата** | `Дата(2024, 1, 15)`, `'20240115120000'` | Точность до секунды. `'00010101000000'` — неопределённая дата |
| **Булево** | `Истина`, `Ложь` | Логический тип |
| **Неопределено** | `Неопределено` | Аналог `null`. Переменная без значения |
| **Тип** | `Тип("Число")`, `ТипЗнч(Значение)` | Ссылка на тип данных |
| **УникальныйИдентификатор** | `Новый УникальныйИдентификатор` | GUID (UUID v4) |

### Объявление переменных

```bsl
// Явное объявление (необязательно, но рекомендуется для читаемости)
Перем Имя, Цена, ДатаПоставки;

// Неявное — через присваивание (тип определяется автоматически)
Наименование = "Товар А";    // Строка
Количество = 10;             // Число
ДатаДок = '20240115';        // Дата
Проведён = Истина;           // Булево
```

**Правило:** используйте `Перем` в начале модуля/процедуры для документирования — так видно, какие переменные используются.

### Определение типа

```bsl
// Тип значения
Если ТипЗнч(Значение) = Тип("Число") Тогда ...

// Проверка на тип
Если Значение instanceof Тип("СправочникСсылка.Номенклатура") Тогда ...

// Проверка на пустоту
Если ПустаяСтрока(Строка) Тогда ...
Если Значение = Неопределено Тогда ...
Если Ссылка.Пустая() Тогда ...

// Универсальная проверка
Если ЗначениеЗаполнено(Значение) Тогда ...   // БСП
```

### Преобразование типов

```bsl
// Строка ← Число/Дата
Строка(1500);                    // "1500"
Формат(ДатаДок, "ДФ=dd.MM.yyyy"); // "15.01.2024"

// Число ← Строка
Число("1500");                   // 1500

// Дата ← Строка
Дата("20240115");                // Дата(2024, 1, 15)
Дата(2024, 1, 15, 12, 0, 0);    // Полная дата

// Булево ← Число
Булево(1);  // Истина
Булево(0);  // Ложь
```

**Правило:** при сравнении значений разных типов платформа **не вызывает ошибку**, но результат может быть неожиданным. Всегда приводите типы явно.

## 5.2) Основные операторы встроенного языка

Основа: книга **И.И. Низамутдинова — «Программирование в 1С за 11 шагов»**, глава 3.

### Условия

```bsl
// Простое условие
Если Количество > 0 Тогда
    Сумма = Количество * Цена;
КонецЕсли;

// С ветвлением
Если Сумма > 10000 Тогда
    Скидка = Сумма * 0.1;
ИначеЕсли Сумма > 5000 Тогда
    Скидка = Сумма * 0.05;
Иначе
    Скидка = 0;
КонецЕсли;

// Тернарный оператор (выражение)
Скидка = ?(Сумма > 10000, 0.1, ?(Сумма > 5000, 0.05, 0));
```

**Правило:** `?()` — это выражение, а не оператор. Все три аргумента **вычисляются** до возврата результата. Избегайте тяжёлых вычислений внутри `?()`.

### Множественный выбор

```bsl
Выбрать
    Когда Статус = Перечисления.Статусы.Черновик
        Тогда Сообщить("Черновик");
    Когда Статус = Перечисления.Статусы.Проведен
        Тогда Сообщить("Проведён");
    Иначе
        Сообщить("Неизвестно");
КонецВыбора;
```

### Циклы

```bsl
// Счётчик
Для Счётчик = 1 По 10 Цикл
    Сообщить(Счётчик);
КонецЦикла;

// С шагом
Для Счётчик = 10 По 1 Цикл
    Сообщить(Счётчик);
КонецЦикла;

// По условию
Пока ЕстьДанные Цикл
    // обработка
КонецЦикла;

// По коллекции
Для Каждого Элемент из Массив Цикл
    Сообщить(Элемент);
КонецЦикла;

// Прервать / Продолжить
Для Каждого Строка из ТЗ Цикл
    Если Строка.Пометка = Ложь Тогда
        Продолжить;  // следующая итерация
    КонецЕсли;
    Если Строка.Стоп Тогда
        Прервать;    // выход из цикла
    КонецЕсли;
КонецЦикла;
```

### Процедуры и функции

```bsl
// Процедура — не возвращает значение
Процедура РассчитатьСкидку(Сумма, Процент)
    Результат = Сумма * Процент / 100;
КонецПроцедуры

// Функция — возвращает значение
Функция РассчитатьСкидкуФ(Сумма, Процент)
    Возврат Сумма * Процент / 100;
КонецФункции
```

**Правила:**
- Параметры передаются **по ссылке** (изменение внутри влияет на оригинал), кроме примитивных типов;
- `Возврат` в процедуре — только выход, без значения;
- Не рекомендуется менять параметры внутри функции — это антипаттерн.

### Попытка / Исключение

```bsl
Попытка
    Объект.Записать();
Исключение
    Сообщить(ОписаниеОшибки());
    ЗаписьЖурналаРегистрации(
        "ЗаписьОбъекта",
        УровеньЖурналаРегистрации.Ошибка,
        ,
        ,
        ОписаниеОшибки()
    );
КонецПопытки;

// Вызов исключения
Если Не ЗначениеЗаполнено(Контрагент) Тогда
    ВызватьИсключение НСтр("ru = 'Не указан контрагент.'");
КонецЕсли;
```

### Ключевые слова (зарезервированные)

Нельзя использовать как имена переменных:

```
Если, Тогда, Иначе, ИначеЕсли, КонецЕсли
Для, По, Цикл, КонецЦикла, Каждого, Из
Процедура, КонецПроцедуры, Функция, КонецФункции
Возврат, Перем, Экспорт
Попытка, Исключение, КонецПопытки, ВызватьИсключение
Новый, Не, Или, И
Выбрать, Когда, ИначеВыбрать, КонецВыбора
Прервать, Продолжить
Выполнить, Прогресс, Ожидать
```

**Именование:** кириллица и латиница допустимы, регистр не имеет значения. Рекомендуется латиница для имён процедур/функций, кириллица допустима в именах переменных для читаемости.

- **Запрос**

- **СКД**
  - понятные параметры и дефолты;
  - группировки/ресурсы соответствуют постановке;
  - расшифровка работает и ведет к нужным формам/данным;
  - интерфейсные тексты локализуемы (см. ИТС `#std762`).

## 10) Рецепты отчетов на СКД (практичные схемы)

### 10.1) «Таблица + Диаграмма» в одном отчете

Подход:
- один набор данных;
- два вывода/две области (табличный документ + диаграмма) или два варианта отчета;
- диаграмму строить по тем же ресурсам/группировкам, что и таблицу, чтобы не было «двух разных отчетов».

Чек:
- ресурсы диаграммы = ресурсы таблицы (или явно объяснено, почему отличаются);
- параметры (период/отбор) общие;
- в заголовках/подписях никаких «зашитых» строк, если они выходят пользователю (см. #std762).

### 10.2) Детализация и «проваливание»

Рекомендуемый путь:
- в основной таблице выводите агрегаты и «ключи» расшифровки;
- расшифровка по клику ведёт:
  - либо на форму списка с отбором,
  - либо на подчинённый отчет‑детализацию,
  - либо на форму объекта (если смысл именно в карточке).

Технический принцип:
- поле, по которому проваливаемся, должно быть **ссылкой/ключом**, а не строковым представлением.

### 10.3) Типовой отчет СКД «Остатки + Обороты + Конечный остаток»

Задача: дать пользователю картину по ресурсу (количество/сумма) за период, с начальным остатком, оборотами и конечным остатком.

Рекомендуемая схема:
- **Параметры**:
  - `ДатаНач`, `ДатаКон` (период),
  - (опционально) `Склад`, `Организация`, `Номенклатура` (отборы).
- **Набор данных**: один запрос на базе виртуальной таблицы **Обороты** регистра, который возвращает:
  - `ОстатокНачальный`,
  - `Приход`,
  - `Расход`,
  - `ОстатокКонечный`.

Ключевая идея: не считать конечный остаток “вручную” на клиенте, а получать его из механизма регистра/виртуальных таблиц.

Скелет запроса (примерно):

```bsl
ТекстЗапроса =
"ВЫБРАТЬ
|   Данные.Номенклатура                          КАК Номенклатура,
|   Данные.Склад                                КАК Склад,
|
|   Данные.КоличествоНачальныйОстаток            КАК НачОстаток,
|   Данные.КоличествоПриход                      КАК Приход,
|   Данные.КоличествоРасход                      КАК Расход,
|   Данные.КоличествоКонечныйОстаток             КАК КонОстаток
|ИЗ
|   РегистрНакопления.ТоварыНаСкладах.Обороты(&ДатаНач, &ДатаКон, , ) КАК Данные
|ГДЕ
|   (&Склад ЕСТЬ NULL ИЛИ Данные.Склад = &Склад)";
```

Настройка СКД:
- **Группировки**: Номенклатура → (опц.) Склад.
- **Ресурсы**: НачОстаток, Приход, Расход, КонОстаток.
- **Расшифровка**:
  - по Номенклатуре/Складу — проваливание в список движений/документов (отчет‑детализация или форма списка документов с отбором по периоду и аналитикам).

## 11) Как работать с синтакс‑помощником (памятка)

Цель — быстро находить **правильные типы/методы/события** и не гадать по памяти.

- **Ищите по “классу” объекта, а не по “как я его назвал в переменной”**:
  - пример: переменная `Выборка` → ищите “Выборка из результата запроса” / “Результат запроса”.
- **Если не знаете, какой это тип**:
  - временно выводите тип в отладке/через `ТипЗнч()` (на время диагностики),
  - затем ищите по названию типа в синтакс‑помощнике.
- **Для UI‑событий ищите по объекту формы/элемента**:
  - “УправляемаяФорма.ПриСозданииНаСервере”,
  - “Расширение поля формы …” (как вы уже видите в комментариях проекта).
- **Для запросов ключевые сущности**:
  - `Запрос` (установка параметров, выполнение),
  - результат выполнения (`Выполнить`, `ВыполнитьПакет`),
  - выборка (`Выбрать`, `Следующий`, `Пустой`).

## 12) Лучшие проверенные практики (ИТС v8std) — «что делать всегда»

Ниже — то, что реально помогает в больших конфигурациях: читабельность, предсказуемость поведения и производительность.

### 12.1) Запросы: базовый минимум качества

- **Оформляйте текст запроса по стандарту** (#std437): `https://its.1c.ru/db/v8std/content/437/hdoc`
- **Проверяйте пустой результат правильно** (#std438): `https://its.1c.ru/db/v8std/content/438/hdoc`
- **Делайте условия эффективными** (точка входа по условиям): `https://its.1c.ru/db/content/v8std/src/300/200/i8100658.htm`
- **Не плодите повторные выполнения однотипных запросов** (#std436): `https://its.1c.ru/db/v8std/content/436/hdoc`

### 12.2) Запросы: «опасные зоны» (то, что нужно избегать без крайней необходимости)

Список ниже — типовые причины тормозов и неожиданных результатов:

- **ПОЛНОЕ ВНЕШНЕЕ СОЕДИНЕНИЕ** — используйте только когда иначе нельзя (#std435): `https://its.1c.ru/db/v8std/content/435/hdoc`
- **ОБЪЕДИНИТЬ / ОБЪЕДИНИТЬ ВСЕ** — осознанно, с пониманием последствий (#std434): `https://its.1c.ru/db/v8std/content/434/hdoc`
- **УПОРЯДОЧИТЬ ПО** — не сортируйте «на всякий случай» (#std412): `https://its.1c.ru/db/v8std/content/412/hdoc`
- **ПОДОБНО (LIKE)** — учитывайте особенности и влияние на производительность (#std726): `https://its.1c.ru/db/v8std/content/726/hdoc`
- **Псевдонимы источников** — делайте псевдонимы осмысленными и единообразными (#std758): `https://its.1c.ru/db/v8std/content/758/hdoc`
- **COUNT и “сколько строк”** — делайте корректно, без лишней нагрузки (#std787): `https://its.1c.ru/db/v8std/content/787/hdoc`

### 12.3) Локализация: запросы/динамические списки/СКД

- **Никаких “зашитых” строк**, которые видит пользователь: выносите в параметры и задавайте локализуемо (#std762): `https://its.1c.ru/db/v8std/content/762/hdoc`

### 12.4) Встроенный язык: базовая дисциплина кода

- Точка входа по общим требованиям к конструкциям встроенного языка: `https://its.1c.ru/db/v8std/content/441/hdoc`
- **Всегда разделяйте ответственность**: данные/бизнес‑логика/представление (клиент) — не смешивайте в одном обработчике «всё сразу».
- **Пишите код так, чтобы его можно было отлаживать**:
  - минимум скрытой динамики (склейки строк, “магии” с неявными типами);
  - предсказуемые точки принятия решений (ранние `Возврат`, явные проверки).

### 12.5) СКД: практики поддерживаемых отчетов

- **Один набор данных** до тех пор, пока не доказана необходимость нескольких.
- **Варианты отчета**:
  - поставляемые варианты — минимальный набор «по ролям»;
  - пользовательские варианты — отдельно, без “ломания” основного.
- **Расшифровка = часть UX**:
  - проваливание должно идти по ссылкам/ключам, а не по представлениям;
  - сценарий «сверху вниз» продуман (от агрегата → к документу/записи).

## 13) Контрольный список «перед тем как сказать "готово"»

- **Запрос**
  - параметры вместо литералов, нет ненужного `УПОРЯДОЧИТЬ ПО`;
  - проверен пустой результат (и понятное поведение UI);
  - условия "эффективные" (нет тяжелых функций над полем в ГДЕ без необходимости);
  - понятные псевдонимы и читаемая структура.
- **Отчет СКД**
  - параметры и дефолты логичны;
  - заголовки/интерфейсные тексты локализуемы;
  - расшифровка ведёт к ожидаемым данным;
  - вариант(ы) отчета соответствует ролям пользователей.

## 14) Транзакции и блокировки (ИТС)

### 14.1) Основные правила работы с транзакциями

См. ИТС:
- Управление транзакциями: `https://its.1c.ru/db/v8std/content/452/hdoc`
- Блокировка данных: `https://its.1c.ru/db/v8std/content/453/hdoc`

**Ключевые принципы:**
- **Минимизируйте время транзакции** — открывайте как можно позже, закрывайте как можно раньше;
- **Не вызывайте интерактивные методы внутри транзакции** (`Предупреждение`, `Вопрос`, `Сообщить`);
- **Не выполняйте длительные вычисления внутри транзакции**;
- **Используйте управляемые блокировки** для оптимистичного контроля:

```bsl
Блокировка = Новый БлокировкаДанных;
Элемент = Блокировка.Добавить("РегистрНакопления.ТоварыНаСкладах");
Элемент.Режим = РежимБлокировкиДанных.Исключительный;
Элемент.УстановитьЗначение("Склад", Склад);
Блокировка.Заблокировать();

НачатьТранзакцию();
Попытка
    // запись данных
    ЗафиксироватьТранзакцию();
Исключение
    ОтменитьТранзакцию();
    ВызватьИсключение;
КонецПопытки;
```

### 14.2) Обработка ошибок и транзакции

**Правило:** всегда используйте конструкцию `Попытка...Исключение` внутри транзакций:

```bsl
НачатьТранзакцию();
Попытка
    // операции с данными
    Записать();
    ЗафиксироватьТранзакцию();
Исключение
    ОтменитьТранзакцию();
    ЗаписьЖурналаРегистрации(
        "ЗаписьДокумента", 
        УровеньЖурналаРегистрации.Ошибка,
        ,
        ,
        ОписаниеОшибки()
    );
    ВызватьИсключение;
КонецПопытки;
```

## 15) Управляемые формы: стандарты и практики (ИТС)

### 15.1) Основные принципы

См. ИТС:
- Построение форм: `https://its.1c.ru/db/v8std/content/573/hdoc`
- Клиент-серверное взаимодействие: `https://its.1c.ru/db/v8std/content/574/hdoc`

**Ключевые правила:**
- **Разделяйте клиентский и серверный код**:
  - `&НаКлиенте` — только UI-логика, реакции на изменения, вызов сервера;
  - `&НаСервере` / `&НаСервереБезКонтекста` — работа с данными, вычисления;
- **Минимизируйте передачу данных между клиентом и сервером**;
- **Используйте `&НаСервереБезКонтекста`** где возможно (быстрее, не требует передачи контекста формы);

```bsl
&НаКлиенте
Процедура КомандаВыполнить(Команда)
    // Только вызов серверного метода и обработка результата
    Результат = СервернаяФункция(Параметр);
    Если Результат Тогда
        Оповестить("ДанныеОбновлены");
    КонецЕсли;
КонецПроцедуры

&НаСервереБезКонтекста
Функция СервернаяФункция(Параметр)
    // Работа с данными без контекста формы
    Запрос = Новый Запрос(ТекстЗапроса);
    Запрос.УстановитьПараметр("Параметр", Параметр);
    Результат = Запрос.Выполнить();
    Возврат Не Результат.Пустой();
КонецФункции
```

### 15.2) Динамические списки

См. ИТС: `https://its.1c.ru/db/v8std/content/580/hdoc`

**Рекомендации:**
- Используйте динамические списки для больших объемов данных (автоматическая подкачка);
- Настраивайте отборы на сервере через `ОбработкаПолученияДанных`:

```bsl
&НаСервере
Процедура СписокПриПолученииДанных(Элемент, Отказ, СтандартнаяОбработка)
    Если СтандартнаяОбработка Тогда
        // Дополнительная обработка запроса
        Запрос = Элемент.ПолучитьЗапрос();
        ТекстЗапроса = Запрос.Текст;
        // Модификация запроса
        Запрос.Текст = ТекстЗапроса;
        Элемент.УстановитьЗапрос(Запрос);
    КонецЕсли;
КонецПроцедуры
```

### 15.3) Команды и действия форм

- **Команды формы** — группируйте логически (основные, дополнительные);
- **Глобальные команды** — только действительно глобальные действия;
- **Контекстные команды** — привязывайте к конкретным элементам;
- **Используйте `ПараметрыКоманды`** для передачи данных в обработчик.

## 16) Регистры: проектирование и использование

### 16.1) Регистры накопления

См. ИТС: `https://its.1c.ru/db/v8std/content/239/hdoc`

**Типовые сценарии:**
- **Остатки** — складской учет, деньги, материалы;
- **Обороты** — продажи, покупки, движения;
- **Остатки и обороты** — комбинированные сценарии.

**Правила:**
- **Измерения** — то, по чему делается срез/группировка (номенклатура, склад, контрагент);
- **Ресурсы** — то, что считается (количество, сумма);
- **Реквизиты** — дополнительная информация (не участвует в агрегации);
- **Виды движений**:
  - `Приход/Расход` — для остатков;
  - `Дебет/Кредит` — для оборотов с двойной записью;

### 16.2) Регистры сведений

См. ИТС: `https://its.1c.ru/db/v8std/content/240/hdoc`

**Типы:**
- **Непериодические** — текущие данные (цены, настройки, соответствия);
- **Периодические** — история изменений (история цен, курсы валют);

**Ведущие измерения:**
- Включайте для автоматического контроля актуальности;
- Пример: регистр "ЦеныНоменклатуры" с ведущим измерением "Номенклатура".

### 16.3) Регистры бухгалтерии

См. ИТС: `https://its.1c.ru/db/v8std/content/238/hdoc`

- Используйте для регламентированного учета;
- Обязательно настраивайте **субконто** для аналитики;
- **План счетов** — основа регистра бухгалтерии.

## 16.4) Объектные типы: Ссылка, Объект, Менеджер

Основа: книга **И.И. Низамутдинова — «Программирование в 1С за 11 шагов»**, главы 6–7.

### 16.4.1) Три уровня работы с объектами метаданных

Каждый объект метаданных (справочник, документ, регистр и т.д.) в 1С существует на трёх уровнях:

| Уровень | Что это | Пример |
|---------|---------|--------|
| **Менеджер** | «Фабрика» объектов — создаёт, ищет, удаляет | `Справочники.Номенклатура` |
| **Ссылка** | Лёгкий указатель на запись в БД (как ID) | `СсылкаНаТовар` |
| **Объект** | Полные данные записи + методы записи/удаления | `ОбъектТовар = Ссылка.ПолучитьОбъект()` |

**Правило:** начинайте со Ссылки. Переходите к Объекту только когда нужно **изменить и записать** данные.

### 16.4.2) Менеджер объекта

Менеджер — точка входа для работы с типом метаданных. Доступен через глобальный контекст:

```bsl
// Менеджер справочника
МенеджерНоменклатуры = Справочники.Номенклатура;

// Менеджер документа
МенеджерРеализации = Документы.РеализацияТоваровУслуг;

// Менеджер перечисления
МенеджерКоробок = Перечисления.КоробкаПередач;
```

**Типовые методы менеджеров:**

| Метод | Возвращает | Описание |
|-------|-----------|----------|
| `НайтиПоНаименованию(Имя, Точное)` | Ссылка | Поиск по наименованию |
| `НайтиПоКоду(Код)` | Ссылка | Поиск по коду |
| `СоздатьЭлемент()` | Объект | Создание нового элемента |
| `Выбрать([Отбор])` | Выборка | Перебор всех элементов |

```bsl
// Создание нового элемента
НовыйОбъект = Справочники.Номенклатура.СоздатьЭлемент();
НовыйОбъект.Наименование = "Товар А";
НовыйОбъект.Артикул = "ART-001";
НовыйОбъект.Записать();

// Получение ссылки
СсылкаНаТовар = НовыйОбъект.Ссылка;
```

### 16.4.3) Ссылка

Ссылка — это **указатель** на конкретную запись в базе данных. Она «лёгкая» (не загружает реквизиты), сериализуется (передаётся клиент↔сервер).

```bsl
// Получение ссылки
Ссылка = Справочники.Номенклатура.НайтиПоНаименованию("Товар А", Истина);

// Проверка на пустоту
Если Ссылка.Пустая() Тогда
    Сообщить("Элемент не найден");
КонецЕсли;

// Получение владельца ссылки (тип метаданных)
Владелец = Ссылка.Метаданные(); // СправочникМенеджер.Номенклатура
```

**Правило:** используйте Ссылку для отборов, сравнений, хранения в реквизитах и параметрах. Не получайте Объект, если нужна только идентификация.

### 16.4.4) Объект (из Ссылки)

Объект — это «развёрнутая» запись со **всеми реквизитами** и методами записи:

```bsl
// Получение объекта из ссылки
ОбъектТовар = СсылкаНаТовар.ПолучитьОбъект();

// Изменение реквизитов
ОбъектТовар.Наименование = "Товар А (обновлённый)";
ОбъектТовар.Записать();

// Обновление ссылки на форме (управляемая форма)
Оповестить("ДанныеОбновлены");
// или
ОбновитьОтображениеДанных(СсылкаНаТовар);
```

**Методы Объекта:**

| Метод | Описание |
|-------|----------|
| `Записать()` | Сохраняет объект в БД |
| `ПолучитьФорму([ИмяФормы])` | Возвращает форму объекта |
| `Ссылка` | Свойство — ссылка на этот же объект |

**События объектов** (модуль объекта):

| Событие | Когда вызывается |
|---------|-----------------|
| `ПередЗаписью(Отказ)` | Перед записью, можно отменить через `Отказ = Истина` |
| `ПриЗаписи(Отказ)` | В момент записи (в транзакции) |
| `ПослеЗаписи(Отказ)` | После записи |
| `ОбработкаПроведения(Отказ, Режим)` | При проведении документа |
| `ОбработкаУдаления(Отказ)` | При удалении элемента |

**Получить события объекта:** откройте модуль объекта → меню **Текст** → **Процедуры и функции**.

### 16.4.5) Выборка

Выборка — объект для **последовательного перебора** записей:

```bsl
Выборка = Справочники.Номенклатура.Выбрать();
Пока Выборка.Следующий() Цикл
    // Выборка — это Ссылка
    Сообщить(Выборка.Наименование);
КонецЦикла;
```

**Важно:** элементы выборки — это **Ссылки**, а не Объекты. Для изменения — вызывайте `ПолучитьОбъект()`:

```bsl
Выборка = Справочники.Номенклатура.Выбрать();
Пока Выборка.Следующий() Цикл
    Если Выборка.ЭтоГруппа = Ложь Тогда
        Объект = Выборка.ПолучитьОбъект();
        Объект.Комментарий = "Обработано";
        Объект.Записать();
    КонецЕсли;
КонецЦикла;
```

### 16.4.6) МенеджерЗаписи (для регистров)

Для регистров сведений используется `МенеджерЗаписи` — альтернатива набору записей:

```bsl
// Создание/изменение записи регистра сведений
Регистр = РегистрыСведений.ЦеныНоменклатуры;
Запись = Регистр.СоздатьМенеджерЗаписи();
Запись.Номенклатура = СсылкаНаТовар;
Запись.ВидЦен = ВидЦен;
Запись.Цена = 1500;
Запись.Период = ТекущаяДата();
Запись.Записать();

// Чтение существующей записи
Запись = Регистр.СоздатьМенеджерЗаписи();
Запись.Номенклатура = СсылкаНаТовар;
Запись.ВидЦен = ВидЦен;
Запись.Прочитать();
Если Запись.Выбран() Тогда
    ТекущаяЦена = Запись.Цена;
КонецЕсли;
```

**Правило:** `МенеджерЗаписи` удобен для одиночных записей. Для пакетных операций используйте **НаборЗаписей** (`СоздатьНаборЗаписей()`).

### 16.4.7) Универсальные коллекции значений

Основа: книга **И.И. Низамутдинова — «Программирование в 1С за 11 шагов»**, глава 7.

Универсальные коллекции значений — объекты встроенного языка для хранения **временных данных в оперативной памяти**. Они:
- Создаются через `Новый`;
- Существуют только в рамках контекста (процедуры, формы, сеанса);
- Уничтожаются при завершении контекста;
- Сериализуются (передаются клиент ↔ сервер).

#### Массив

Упорядоченная коллекция с доступом по **индексу** (нумерация с 0). Элементы — любого типа.

```bsl
// Создание
Массив = Новый Массив(5);        // 5 элементов (тип Неопределено)
ПустойМассив = Новый Массив;     // пустой массив

// Заполнение
Массив[0] = "Первый";
Массив[1] = 2;
Массив[2] = ТекущаяДата();

// Добавление
Массив.Добавить("Новый");         // в конец
Массив.Вставить(1, "Второй");    // по индексу

// Чтение
Значение = Массив[0];

// Обход
Для Каждого Элемент из Массив Цикл
    Сообщить(Элемент);
КонецЦикла;

// Удаление
Массив.Удалить(0);               // по индексу
Массив.Очистить();               // все элементы

// Полезные методы
Массив.Количество();              // число элементов
Массив.ВГраница();                // максимальный индекс (Количество - 1)
Массив.Найти(Значение);           // индекс элемента или Неопределено
```

**Многомерный массив:**

```bsl
// Двумерный массив 3x3
Матрица = Новый Массив(3, 3);
Матрица[0][0] = 1;
Матрица[1][2] = "Значение";
```

**Правило:** используйте `Для Каждого` вместо перебора по индексам — код чище и безопаснее.

#### Структура

Коллекция пар **Ключ–Значение**. Ключ — строка, уникален в рамках структуры. Доступ как к свойствам объекта.

```bsl
// Создание
СтруктураТовара = Новый Структура;
СтруктураТовара.Вставить("Номенклатура", СсылкаНаТовар);
СтруктураТовара.Вставить("Количество", 10);
СтруктураТовара.Вставить("Цена", 1500);

// Создание с заполнением (конструктор)
СтруктураТовара = Новый Структура("Номенклатура, Количество, Цена",
    СсылкаНаТовар, 10, 1500);

// Чтение
Товар = СтруктураТовара.Номенклатура;
Цена = СтруктураТовара.Цена;

// Или через свойство
Цена = СтруктураТовара.Свойство("Цена");

// Проверка наличия ключа
Если СтруктураТовара.Свойство("Скидка") Тогда
    Скидка = СтруктураТовара.Скидка;
КонецЕсли;

// Перебор
Для Каждого КлючИЗначение из СтруктураТовара Цикл
    Сообщить(КлючИЗначение.Ключ + " = " + КлючИЗначение.Значение);
КонецЦикла;
```

**Правило:** Структура — лучший способ передать несколько параметров в серверный метод.

#### Соответствие

Аналог структуры, но ключ — **любого типа** (не только строка). Доступ через квадратные скобки.

```bsl
// Создание
Соответствие = Новый Соответствие;

// Заполнение (ключ — любой тип)
Соответствие["Наименование"] = "Товар А";
Соответствие[СсылкаНаТовар] = 1500;          // ссылка как ключ
Соответствие[1] = "Первый";                   // число как ключ

// Чтение
Значение = Соответствие["Наименование"];
Цена = Соответствие[СсылкаНаТовар];

// Удаление
Соответствие.Удалить("Наименование");

// Перебор
Для Каждого КлючИЗначение из Соответствие Цикл
    Сообщить(КлючИЗначение.Ключ + " => " + КлючИЗначение.Значение);
КонецЦикла;
```

| Структура | Соответствие |
|-----------|-------------|
| Ключ — только строка | Ключ — любой тип |
| Доступ: `Структура.Ключ` | Доступ: `Соответствие["Ключ"]` |
| Быстрее | Чуть медленнее, но гибче |

#### ТаблицаЗначений

Таблица в памяти с колонками и строками. Аналог временной таблицы, но **не в БД**.

```bsl
// Создание с колонками
ТЗ = Новый ТаблицаЗначений;
ТЗ.Колонки.Добавить("Номенклатура", ОписаниеТипов("СправочникСсылка.Номенклатура"));
ТЗ.Колонки.Добавить("Количество", ОписаниеТипов("Число", 15, 3));
ТЗ.Колонки.Добавить("Цена", ОписаниеТипов("Число", 15, 2));
ТЗ.Колонки.Добавить("Сумма", ОписаниеТипов("Число", 15, 2));

// Добавление строк
Строка = ТЗ.Добавить();
Строка.Номенклатура = СсылкаНаТовар;
Строка.Количество = 10;
Строка.Цена = 1500;
Строка.Сумма = 15000;

// Заполнение из запроса
ТЗ = Запрос.Выполнить().Выгрузить();

// Перебор
Для Каждого Строка из ТЗ Цикл
    Сообщить(Строка.Номенклатура + ": " + Строка.Сумма);
КонецЦикла;

// Поиск
НайденнаяСтрока = ТЗ.Найти(СсылкаНаТовар, "Номенклатура");

// Сортировка
ТЗ.Сортировать("Сумма Убыв, Номенклатура Возр");

// Свёртка (агрегация)
ТЗ.Свернуть("Номенклатура", "Количество, Сумма");

// Копирование
КопияТЗ = ТЗ.Скопировать();

// Выгрузка колонки в массив
МассивЦен = ТЗ.ВыгрузитьКолонку("Цена");
```

**Правило:** `ТаблицаЗначений` — основной «буфер» между запросом и СКД/печатной формой.

#### ДеревоЗначений

Иерархическая таблица (родитель → дети). Используется для деревьев в формах.

```bsl
Дерево = Новый ДеревоЗначений;
Дерево.Колонки.Добавить("Наименование");
Дерево.Колонки.Добавить("Сумма");

// Корневая строка
Корень = Дерево.Строки.Добавить();
Корень.Наименование = "Группа А";

// Дочерняя строка
Дочерняя = Корень.Строки.Добавить();
Дочерняя.Наименование = "Товар 1";
Дочерняя.Сумма = 1500;
```

#### Фиксированные коллекции

Неизменяемые версии коллекций. Попытка изменить вызовет ошибку.

```bsl
ФиксМассив = Новый ФиксированныйМассив(Массив);
ФиксСтруктура = Новый ФиксированнаяСтруктура(Структура);
ФиксСоответствие = Новый ФиксированноеСоответствие(Соответствие);
```

**Применение:** передача данных в код, который не должен их модифицировать (защита от побочных эффектов).

## 17) Документы: проведение и движения

### 17.1) Модели проведения

См. ИТС: `https://its.1c.ru/db/v8std/content/252/hdoc`

**Три модели:**
1. **Оперативная** (по умолчанию) — движения в момент записи:
   ```bsl
   Процедура ОбработкаПроведения(Отказ, Режим)
       // Формирование движений
   КонецПроцедуры
   ```

2. **Оперативная с проведением** — разделение записи и проведения:
   ```bsl
   Процедура ОбработкаПроведения(Отказ, РежимПроведения)
       Если РежимПроведения.РежимПроведения = РежимПроведенияДокумента.Оперативный Тогда
           // Оперативное проведение
       КонецЕсли;
   КонецПроцедуры
   ```

3. **Перепроведение** — для пакетных операций:
   ```bsl
   Процедура ОбработкаПроведения(Отказ, РежимПроведения)
       Если РежимПроведения.РежимПроведения = РежимПроведенияДокумента.Перепроведение Тогда
           // Удаление старых движений
           Движения.Очистить();
       КонецЕсли;
       // Формирование новых движений
   КонецПроцедуры
   ```

### 17.2) Движения документов

**Правила:**
- **Заполняйте регистратор** — ссылка на документ автоматически;
- **Контролируйте отрицательные остатки**:
  ```bsl
  Процедура ОбработкаПроведения(Отказ, Режим)
      Режим.ДопускатьОтрицательныеОстатки = Ложь;
      // Движения
  КонецПроцедуры
  ```
- **Используйте `Движения.Очистить()`** при перепроведении.

## 18) Общие модули: проектирование и использование

См. ИТС: `https://its.1c.ru/db/v8std/content/431/hdoc`

### 18.1) Типы общих модулей

| Тип модуля | Вызов сервера | Вызов клиента | Внешний контекст | Сервер, без контекста |
|-----------|---------------|---------------|------------------|----------------------|
| **Серверный** | ✓ | | | ✓ |
| **Клиентский** | | ✓ | | |
| **Серверный вызов** | ✓ | ✓ | ✓ | |
| **Без контекста** | ✓ | | | ✓ |

### 18.2) Правила именования и группировки

- **`ОбщепМодуль<Назначение>`** — по функциональной области:

# Стандарты разработки на основе БСП (Библиотека Стандартных Подсистем)

> Раздел основан на анализе БСП редакции 3.1 (версия 3.1.10.441)

## 25) Модульная архитектура: система суффиксов

БСП использует строгую систему именования общих модулей по суффиксам. Это обеспечивает:
- Понятное назначение каждого модуля
- Корректное разделение клиент/сервер
- Возможность расширения без модификации ядра

### 25.1) Суффиксы модулей

| Суффикс | Назначение | Флаги | Пример |
|---------|-----------|-------|--------|
| *(без суффикса)* | Основной серверный API подсистемы | Сервер, ВнешнееСоединение, ВызовСервера | `Пользователи`, `УправлениеДоступом`, `РаботаСФайлами` |
| `Служебный` | Внутренняя реализация, не для внешних вызовов | Сервер | `ВариантыОтчетовСлужебный`, `АдресныйКлассификаторСлужебный` |
| `СлужебныйКлиент` | Служебный код на клиенте | Клиент | `ВариантыОтчетовСлужебныйКлиент` |
| `СлужебныйКлиентСервер` | Служебный код общего назначения | Клиент, Сервер | `ВариантыОтчетовСлужебныйКлиентСервер` |
| `СлужебныйВызовСервера` | Обёртки для серверных вызовов из служебного кода | Сервер, ВызовСервера | `ВариантыОтчетовСлужебныйВызовСервера` |
| `СлужебныйПовтИсп` | Служебный модуль повторного использования | Сервер | `АдресныйКлассификаторПовтИсп` |
| `Клиент` | Клиентская часть подсистемы | Клиент | `ВариантыОтчетовКлиент`, `УправлениеПечатьюКлиент` |
| `КлиентСервер` | Модуль общего назначения (клиент + сервер) | Клиент, Сервер | `СтроковыеФункцииКлиентСервер`, `ДлительныеОперацииКлиентСервер` |
| `ВызовСервера` | Обёртки серверных вызовов для клиентского кода | Сервер, ВызовСервера | `БизнесПроцессыИЗадачиВызовСервера` |
| `Переопределяемый` | Модуль с хуками для расширения другими разработчиками | Сервер, ВнешнееСоединение | `УправлениеДоступомПереопределяемый` |
| `КлиентПереопределяемый` | Клиентские переопределяемые процедуры | Клиент | `БизнесПроцессыИЗадачиКлиентПереопределяемый` |
| `Глобальный` | Глобальный модуль (доступен без префикса) | Глобальный | `ЦентрМониторингаГлобальный` |
| `Локализация` | Региональная локализация (РФ) | Клиент, Сервер | `СтроковыеФункцииКлиентСерверЛокализация` |

### 25.2) Типовой набор модулей подсистемы

Для каждой подсистемы БСП создаёт 5–12 модулей:

```
<Подсистема>                     — серверный API (экспортные процедуры/функции)
<Подсистема>Клиент               — клиентская логика (команды формы, реакции на события)
<Подсистема>КлиентСервер         — общие функции, доступные и клиенту, и серверу
<Подсистема>ВызовСервера         — обёртки серверных вызовов для клиента
<Подсистема>Служебный            — внутренняя реализация (не экспортный)
<Подсистема>СлужебныйКлиент      — клиентские служебные процедуры
<Подсистема>СлужебныйКлиентСервер — общие служебные функции
<Подсистема>СлужебныйВызовСервера — служебные серверные вызовы
<Подсистема>СлужебныйПовтИсп     — код для переиспользования другими подсистемами
<Подсистема>Переопределяемый     — хуки: пустые процедуры для переопределения
<Подсистема>КлиентПереопределяемый — клиентские хуки
```

**Правило:** не создавайте «god-модули» — логику разделяйте по назначению и по клиент/сервер.

### 25.3) Переопределяемые модули (механизм хуков)

БСП позволяет расширять поведение без модификации ядра через переопределяемые модули:

```bsl

// Переопределяемая процедура: вызывается после проверки прав
Процедура ПослеПроверкиВключенияПользователяВГруппу(Пользователь, Группа) Экспорт
    // Пустая реализация — переопределяется в прикладном решении
КонецПроцедуры

// Переопределяемая функция: позволяет добавить дополнительные ограничения
Функция ПроверитьДополнительныеОграничения(Пользователь, Объект) Экспорт
    Возврат Истина; // По умолчанию — без ограничений
КонецФункции
```

**Правило:** если ваша конфигурация расширяет БСП, создавайте свои переопределяемые модули по тому же паттерну.

# 39) Антипаттерны и известные ошибки генерации XML метаданных

> Раздел создан на основе реальных ошибок, допущенных при генерации конфигурации для платформы 1С:Предприятие 8.3.27.
> **Всегда проверять эти пункты перед записью XML-файлов.**

## 39.1) Формы (Form)

**Ошибка:** Добавление свойств `ExtendedPresentation` и `Explanation` в объект `Form`.
**Правило:** Эти свойства **не входят** в состав метаданных формы. Удалять их при генерации XML форм.

```xml
<!-- НЕПРАВИЛЬНО -->
<Form ...>
    <Properties>
        ...
        <ExtendedPresentation/>
        <Explanation/>
    </Properties>
</Form>

<!-- ПРАВИЛЬНО -->
<Form ...>
    <Properties>
        <Name>...</Name>
        <Synonym>...</Synonym>
        <FormType>Managed</FormType>
        <!-- ExtendedPresentation и Explanation ОТСУТСТВУЮТ -->
    </Properties>
</Form>
```

## 39.2) Табличные части справочников (Catalog.TabularSection)

**Ошибка:** Отсутствие узла `InternalInfo` для табличной части в справочнике.
**Правило:** Платформа 8.3.27 требует наличия `InternalInfo` с типами `CatalogTabularSection` и `CatalogTabularSectionRow` для каждой табличной части справочника.

```xml
<!-- ПРАВИЛЬНО -->
<TabularSection uuid="...">
    <InternalInfo>
        <xr:GeneratedType name="CatalogTabularSection.Справочник.ИмяТЧ" category="TabularSection">
            <xr:TypeId>...</xr:TypeId>
            <xr:ValueId>...</xr:ValueId>
        </xr:GeneratedType>
        <xr:GeneratedType name="CatalogTabularSectionRow.Справочник.ИмяТЧ" category="TabularSectionRow">
            <xr:TypeId>...</xr:TypeId>
            <xr:ValueId>...</xr:ValueId>
        </xr:GeneratedType>
    </InternalInfo>
    <Properties>...</Properties>
</TabularSection>
```

## 39.3) Атрибуты табличных частей (TabularSection.Attribute)

**Ошибка:** Добавление свойств `FillValue`, `FillFromFillingValue`, `ChoiceFoldersAndItems`, `Indexing` в атрибуты табличных частей.
**Правило:** Атрибуты ТЧ поддерживают только `Name`, `Synonym`, `Comment`, `Type`. Остальные свойства (заполнение, выбор, индексация) **недопустимы**.

```xml
<!-- НЕПРАВИЛЬНО -->
<Attribute uuid="...">
    <Properties>
        <Name>Поле</Name>
        <Type>...</Type>
        <FillValue xsi:type="xs:string"/> <!-- ОШИБКА -->
        <Indexing>Index</Indexing>        <!-- ОШИБКА -->
    </Properties>
</Attribute>

<!-- ПРАВИЛЬНО -->
<Attribute uuid="...">
    <Properties>
        <Name>Поле</Name>
        <Type>...</Type>
    </Properties>
</Attribute>
```

## 39.4) Справочники (Catalog)

**Ошибка:** Использование `xr:ObjectId` вместо `xr:ValueId` в `InternalInfo`.
**Правило:** Для версии платформы 8.3.27 использовать `xr:ValueId` для идентификаторов типов (Object, Ref, Selection, List, Manager).

## 39.5) План видов характеристик (ChartOfCharacteristicTypes)

**Ошибка:** Хранение свойств объектов в табличных частях справочников строковыми значениями (`ТипСвойства` как `xs:string`).
**Правило:** Для типизированных свойств использовать механизм `ChartOfCharacteristicTypes` (План видов характеристик) БСП:

```xml
<!-- ChartOfCharacteristicTypes.ВидыХарактеристик -->
<ChartOfCharacteristicTypes uuid="...">
    <Properties>
        <Name>ВидыХарактеристик</Name>
        <CharacteristicType>CatalogRef.Характеристики</CharacteristicType>
        <!-- Определяет, какой справочник хранит значения характеристик -->
    </Properties>
</ChartOfCharacteristicTypes>

<!-- Справочник Характеристики: вместо строки — ссылка на план видов -->
<Attribute uuid="...">
    <Properties>
        <Name>ВидХарактеристики</Name>
        <Type>
            <v8:Type>cfg:ChartOfCharacteristicTypesRef.ВидыХарактеристик</v8:Type>
        </Type>
    </Properties>
</Attribute>
```

**Архитектура свойств по БСП:**
| Объект | Назначение |
|---|---|
| `ChartOfCharacteristicTypes.ВидыХарактеристик` | Типы свойств: "Цвет" (строка), "Размер" (строка), "Вес" (число) |
| `Catalog.ЗначенияСвойствОбъектов` | Конкретные значения: "Красный", "XL", "5.2 кг" |
| `InformationRegister.ДополнительныеСведения` | Связь: Объект + Свойство → Значение |

## 39.6) Общие правила валидации

1. **Сверять структуру с примерами из папки `Examp`** перед генерацией нового объекта.
2. **Не копировать слепо свойства** из одного типа объекта в другой (например, свойства реквизита справочника не подходят для атрибута табличной части).
3. **Генерировать уникальные UUID** через PowerShell `[guid]::NewGuid()`, не использовать заглушки вроде `11111111-...` или `aaaaaaaa-...` для финальных файлов.

## 39.7) `Объект` вместо `ЭтотОбъект` в модуле объекта

**Ошибка:** В модуле объекта документа/справочника использовалось `Объект.Реквизит`, что вызывало ошибку *«Переменная не определена (Объект)»*.

**Причина:** `Объект` — это предопределённая переменная **модуля формы**, а не модуля объекта. В модуле объекта нужно использовать `ЭтотОбъект`.

**Правило:**
| Контекст | Предопределённая переменная |
|----------|----------------------------|
| Модуль объекта (справочник, документ, регистр) | `ЭтотОбъект` |
| Модуль формы | `Объект` (ссылается на объект за формой) |

## 39.8) Поле в `InputByString` без индексации

**Ошибка:** При указании кастомного реквизита (например, `Артикул`) в `InputByString` платформа выдаёт ошибку *«Указано неверное поле для ввода по строке»*.

**Причина:** Для того чтобы поле было доступно в `InputByString`, у него должно быть `<Indexing>Index</Indexing>`. Без индекса платформа не может использовать поле для строкового поиска.

**Правило:** если реквизит включается в `InputByString`, обязательно устанавливать `<Indexing>Index</Indexing>`.

## 39.9) Отсутствие `Characteristic` в `InternalInfo` плана видов характеристик

**Ошибка:** Платформа выдаёт ошибку *«отсутствует один или более типов объекта ChartOfCharacteristicTypes»* при загрузке файла плана видов характеристик.

**Причина:** В `InternalInfo` отсутствует элемент `<xr:GeneratedType name="Characteristic.Имя" category="Characteristic">`. Этот тип обязателен — он определяет тип значения характеристики.

**Правило:** в `InternalInfo` плана видов характеристик всегда должны быть **6 типов**: `Object`, `Ref`, `Selection`, `List`, `Characteristic`, `Manager`.

## 39.10) Свойства справочника в плане видов характеристик

**Ошибка:** В файл `ChartOfCharacteristicTypes.xml` были добавлены свойства, которые принадлежат только `Catalog`: `CodeType`, `CharacteristicType`, `Owners`. Значение `WholeCatalog` для `CodeSeries` также неверно.

**Причина:** `ChartOfCharacteristicTypes` не имеет свойств `CodeType`, `CharacteristicType`, `Owners`. Свойство `CodeSeries` есть, но с допустимым значением `WholeCharacteristicKind` (а не `WholeCatalog`).

**Правило:** не копировать свойства из `Catalog` в `ChartOfCharacteristicTypes`. У плана видов характеристик `CodeSeries = WholeCharacteristicKind`.

## 39.11) Отсутствие `ChildObjects` в плане видов характеристик

**Ошибка:** Платформа выдаёт ошибку *«Текущее ChartOfCharacteristicTypes, ожидаемое ChildObjects»*.

**Причина:** Узел `<ChildObjects/>` обязателен для `ChartOfCharacteristicTypes`, даже если у него нет подчинённых объектов (реквизитов, табличных частей).

**Правило:** всегда добавлять пустой `<ChildObjects/>` в конец `ChartOfCharacteristicTypes`.

## 39.12) Лишний план видов характеристик вместо табличной части

**Ошибка:** Для хранения свойств «Цвет», «Размер» у справочника «Характеристики» был создан `ChartOfCharacteristicTypes.ВидыХарактеристик`, хотя по ТЗ нужен простой справочник, подчинённый «Номенклатуре», с табличной частью «Свойства» (колонки: `ТипСвойства`, `Значение`).

**Причина:** План видов характеристик используется, когда типы свойств заранее неизвестны и могут добавляться пользователями. Если типы свойств фиксированы («Цвет», «Размер»), достаточно табличной части в справочнике.

**Правило:** если ТЗ требует «создать 2 свойства — цвет, размер» — это табличная часть справочника, а не план видов характеристик. План видов характеристик нужен только для **произвольных**, настраиваемых пользователем свойств.

## 39.13) ТипСвойства как строка вместо ссылки

**Ошибка:** В табличной части «Свойства» справочника «Характеристики» колонка `ТипСвойства` была типа `xs:string`. По правильной архитектуре (Infostart, БСП) это должна быть **ссылка на справочник типов свойств**.

**Причина:** Строковый тип не обеспечивает ссылочной целостности и не позволяет централизованно управлять типами свойств.

**Правило:** архитектура характеристик по БСП/Infostart:
- **Справочник «ТипыСвойств»** — справочник с предопределёнными элементами «Цвет», «Размер» и реквизитом `ТипЗначения`
- **Справочник «Характеристики»** (подчинён «Номенклатуре») — содержит ТЧ «Свойства»:
  - `ТипСвойства` → `CatalogRef.ТипыСвойств`
  - `Значение` → строка (или составной тип)

## 39.14) PredefinedItem внутри основного XML справочника

**Ошибка:** Элементы `<PredefinedItem>` были добавлены прямо в `ChildObjects` основного XML-файла справочника, что вызывало ошибку *«PredefinedItem не является подчиненным для объекта Catalog»*.

**Причина:** Предопределённые данные справочника хранятся не в основном файле метаданных, а в отдельном файле `Ext/Predefined.xml` внутри папки справочника.

**Правило:** для предопределённых элементов справочника нужно:
1. Создать папку `Catalogs/<ИмяСправочника>/Ext/`
2. Создать файл `Predefined.xml` с содержимым типа `CatalogPredefinedItems`
3. В основном XML справочника **не указывать** `<PredefinedItem>` в `ChildObjects`.

## 39.15) Скрытие автоматически заполняемого поля на форме

**Ошибка:** Поле «Наименование» показывалось пользователю на форме, хотя заполнялось автоматически из значений свойств. Это вызывало путаницу и ошибку «Поле Наименование не заполнено».

**Правило:** если поле заполняется автоматически в `ПередЗаписью`, его нужно скрыть на форме:
1. Создать форму элемента справочника
2. В модуле формы `ПриСозданииНаСервере` установить `Элементы.Наименование.Видимость = Ложь`

## 39.16) Код предопределённого элемента кириллицей

**Ошибка:** Код предопределённого элемента был задан кириллицей (`Размер`, `Цвет`), что противоречит стандартам БСП/УНФ.

**Правило:** код предопределённого элемента — **латиницей** (транслитерация). Наименование (`Description`) — кириллицей для пользователя.
Пример: `Name=Color, Code=Color, Description=Цвет`.

## 39.17) Отсутствие формы списка справочника

**Ошибка:** При нажатии на ссылку справочника в командном интерфейсе программа закрывалась (падала).

**Причина:** У справочника не было формы списка (`ФормаСписка`). Платформа пытается открыть форму списка по умолчанию, не находит её и падает.

**Правило:** каждый справочник, доступный из командного интерфейса, должен иметь:
- `ФормаСписка` (форма списка) — обязательна
- `ФормаЭлемента` (форма элемента) — обязательна для редактирования

## 39.18) Форма вне ChildObjects в справочнике

**Ошибка:** Элемент `<Form>` был размещён после закрывающего тега `</ChildObjects>`, что вызывало ошибку *«Текущее Form, ожидаемое Catalog»*.

**Правило:** в XML справочника элемент `<Form>` должен находиться **внутри** `<ChildObjects>`, после всех реквизитов и табличных частей, но **перед** закрывающим тегом `</ChildObjects>`.

## 39.19) Автозаполнение обязательного «Наименование» только для новых

**Ошибка:** Код в `ПередЗаписью` заполнял `Наименование` только если `ЭтоНовый()`. При редактировании существующей характеристики без свойств платформа выдавала *«Поле Наименование не заполнено»*.

**Правило:** если `Наименование` заполняется автоматически, код в `ПередЗаписью` должен заполнять его **всегда**, с fallback-значением:
```bsl
Если Значения.Количество() > 0 Тогда
    Наименование = СтрСоединить(Значения, ", ");
Иначе
    Наименование = "Характеристика " + ЭтотОбъект.Код;
КонецЕсли;
```

## 39.20) IncludeInCommandInterface в справочнике

**Ошибка:** Свойство `IncludeInCommandInterface` было добавлено в XML справочника, что вызывало ошибку *«Свойство IncludeInCommandInterface не входит в состав объекта метаданных Catalog»*.

**Причина:** В некоторых версиях платформы 1С это свойство не поддерживается для объектов типа `Catalog`.

**Правило:** для скрытия справочника из командного интерфейса не использовать `IncludeInCommandInterface` в XML. Подчинённые справочники автоматически не отображаются в главном меню. Если нужно явно скрыть — это настраивается через состав командного интерфейса в конфигураторе.

## 40.10) Практические правила разработки в УНФ

### Добавление нового реквизита в справочник/документ
1. Создать реквизит в метаданных.
2. Добавить в `ДополнительныеРеквизиты` табличную часть (если универсальный).
3. Если нужен типизированный — включить в `ChartOfCharacteristicTypes.ДополнительныеРеквизитыИСведения`.
4. Значения хранить в `InformationRegister.ДополнительныеСведения`.

### Добавление нового движения документа
1. Проверить существующие регистры накопления (243+).
2. Не создавать дубликаты — использовать `Запасы`, `ВзаиморасчетыСКонтрагентами`, `ДенежныеСредства` и т.д.
3. В `ОбработкаПроведения` очищать движения при `Перепроведение`.

### Создание нового отчета
1. Использовать СКД (Систему компоновки данных).
2. Базовая форма: `CommonForm.ФормаОтчета`.
3. Настройки: `CommonForm.ФормаНастроекОтчета`.
4. Включить в подсистему `ВариантыОтчетов`.

### Создание новой обработки
1. Проверить `CommonForm.ДополнительныеОтчетыИОбработки` — возможно, можно подключить как внешнюю.
2. Для длительных операций: `CommonForm.ДлительнаяОперация` + модуль `ДлительныеОперации`.

### Расширение функциональности
1. Проверить `Переопределяемый` модули — они специально для расширения.
2. Следовать архитектуре БСП: не модифицировать ядро, а переопределять хуки.

## 41.5) Обработка проведения типового документа УНФ

```bsl
// Типовая структура обработки проведения в УНФ
Процедура ОбработкаПроведения(Отказ, РежимПроведения)
    
    // 1. Очистка движений при перепроведении
    Если РежимПроведения.РежимПроведения = РежимПроведенияДокумента.Перепроведение Тогда
        Движения.Запасы.Очистить();
        Движения.ВзаиморасчетыСКонтрагентами.Очистить();
        Движения.ДенежныеСредства.Очистить();
    КонецЕсли;
    
    // 2. Формирование движений по товарам
    ЗаполнитьДвиженияЗапасы();
    
    // 3. Формирование движений по взаиморасчетам
    ЗаполнитьДвиженияВзаиморасчеты();
    
    // 4. Запись движений
    Движения.Записать();
    
КонецПроцедуры

Процедура ЗаполнитьДвиженияЗапасы()
    
    Для Каждого ТекСтрока Из ЭтотОбъект.Товары Цикл

        Движение = Движения.Запасы.Добавить();
        Движение.ВидДвижения = ВидДвиженияНакопления.Приход; // или Расход
        Движение.Период = ЭтотОбъект.Дата;
        Движение.Регистратор = ЭтотОбъект.Ссылка;
        Движение.Номенклатура = ТекСтрока.Номенклатура;
        Движение.Склад = ЭтотОбъект.Склад;
        Движение.Количество = ТекСтрока.Количество;
        Движение.Сумма = ТекСтрока.Сумма;
        
    КонецЦикла;
    
КонецПроцедуры
```
